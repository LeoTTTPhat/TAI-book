\chapter{Socio-Technical Trust and Human Agency}
\label{chap:hitl}

\abstract{True trustworthiness exists at the intersection of technical reliability and human agency. By 2026, the industry has realized that a system providing perfectly private answers remains a failure if it lacks "Social Legibility" and a clear hierarchy of control. This chapter explores the evolution of Socio-Technical Trust, specifically the regulatory and architectural shift toward Human-on-the-Loop (HOTL) and Human-in-Command (HiC) models. We analyze the technical implementation of "The Emergency Brake," explore the move toward rigorous Conformal Prediction for uncertainty quantification, and provide a 2026 case study on autonomous healthcare triage.}

\section{The Interdependency of Trust}
\label{sec:interdependency_trust}

In 2026, the ``Trust Gap'' has become the primary bottleneck for AI adoption. This gap is the distance between what an autonomous agent \textit{can} do and what a human operator \textit{understands} it is doing. We have moved beyond the ``Black Box'' era into one where \textbf{Social Legibility} is a technical requirement.

Trust is highest when two layers are harmonized:
\begin{description}
  \item[Technical Trust:] The model's inherent stability, resilience, and mathematical privacy guarantees (e.g., $\epsilon$-DP, robust recovery).
  \item[Social Trust:] The presence of ``Institutional Backstops''---regulations, audit trails, and the assurance that a human or organization remains responsible for the agent's actions.
\end{description}

\section{HITL, HOTL, and Human-in-Command (HiC)}
\label{sec:agency_spectrum}

The 2026 regulatory environment, specifically Vietnam's AI Law and the updated NIST framework, defines a spectrum of human agency that architects must implement:

\begin{description}
  \item[Human-in-the-Loop (HITL):] Active human participation in every decision (e.g., a doctor approving every individual prescription from an AI).
  \item[Human-on-the-Loop (HOTL):] The agent acts autonomously, but a human monitors the stream and can intervene or override in real-time.
  \item[Human-in-Command (HiC):] A high-level governing requirement where humans set the boundaries, constitutional constraints, and ``Emergency Brakes'' within which the agent operates.
\end{description}

Architecturally, the \textbf{Emergency Brake Pattern} is a hardcoded circuit-breaker that immediately suspends agent execution if a core safety or privacy boundary (defined in the Norm Matrix from Chapter 2) is approached.

\subsection{The Post-Brake Recovery Protocol}
\label{sec:post-brake-recovery}

When the Emergency Brake trips, the agent enters the \textsc{Suspended}
state of the Confidence Spectrum (Section~\ref{sec:sociotechnical-oversight}).
A critical failure in many early autonomous architectures was neglecting the
\emph{restart mechanism}: simply un-pausing the agent often resulted in an
immediate re-triggering of the brake, trapping the system in an infinite
failure loop—the scenario analysed in the Practitioner's Challenge of
Section~\ref{sec:trenches-ch3}.

To resolve this, the Architect must implement a rigorous
\textbf{Human-in-Command (HiC) Transition Protocol} consisting of four
phases, each with explicit entry conditions and exit criteria that prevent
premature re-engagement of the autonomous loop.

% ------------------------------------------------------------------
\subsubsection*{Phase 0: Brake Acquisition and State Isolation}
% ------------------------------------------------------------------

Before any human deliberation can begin, the system must guarantee that the
agent's causal environment is frozen.  Three actions occur atomically upon
brake activation:

\begin{enumerate}
  \item \textbf{Context Freezing and State Dump.}
        The precise memory state, the active tool-call queue, and the
        specific sequence of LLM generation steps that triggered the brake
        are cryptographically signed and written to the ZK-Audit Log
        (Chapter~\ref{ch:zk-ml}).  The agent is entirely sequestered from
        new inputs: no new user turns, no tool responses, no background
        memory retrievals are admitted to the context window.

  \item \textbf{Causal Chain Annotation.}
        The monitoring layer annotates the frozen state with the
        \emph{exact predicate} or Constitutional Policy clause
        (Section~\ref{sec:constitutional-alignment}) that the agent
        attempted to violate.  This annotation—a deterministic Datalog
        fact, not a probabilistic score—is the primary artefact passed
        to the human supervisor.  Without it, the supervisor is forced to
        reconstruct intent from raw logs, a process that is both slow and
        error-prone under operational pressure.

  \item \textbf{DID Suspension.}
        If the deployment uses Decentralised Identifiers
        (Section~\ref{sec:agentic-identity}), the agent's DID is
        immediately flagged as \texttt{suspended} and a Body-Snatch Alert
        is filed in the Trust Debt Registry
        (Section~\ref{sec:trust-debt-registry}), ensuring that no
        authenticated API call can be issued by the agent during the
        deliberation window.
\end{enumerate}

% ------------------------------------------------------------------
\subsubsection*{Phase 1: Human Deliberation via the Trust Dashboard}
% ------------------------------------------------------------------

The Trust Dashboard (Section~\ref{sec:trust-dashboards}) now renders the
\textsc{Suspended}-state view—distinct from the \textsc{Handover} panel of
Design Pattern~3.  Where the Handover panel asks the operator to
\emph{choose} among candidate recommendations, the Suspension panel asks the
operator to \emph{diagnose} a constitutional failure.

The Suspension panel exposes exactly three information zones:

\begin{description}
  \item[Violation Zone (top, red border).]
        A plain-language rendering of the violated predicate.  Example:
        \textit{``The agent attempted to transfer patient data to
        \texttt{InsuranceAPI\_SG} without DPO authorisation
        (Decree~13, Art.~26).''}
        No probability scores, no model internals—only the
        violated constitutional fact.

  \item[Context Diff Zone (centre).]
        A structured ``before / after'' view of the agent's context
        window at the moment of brake activation, with the offending
        tool-call parameters highlighted.  The diff is rendered against
        the last \emph{known-safe} checkpoint to make the causal delta
        immediately legible.

  \item[Mutation Control Zone (bottom).]
        The four and only four actions the supervisor may take, each of
        which \emph{mutates} the agent state before re-engagement
        (see Phase~2):
        \begin{enumerate}
          \item Inject a corrective system prompt into the context window.
          \item Revoke a specific tool from the active tool registry.
          \item Clear the tool-call queue and reset working memory.
          \item Escalate to the Senior Operator (advancing the
                Graceful Degradation Ladder, Design Pattern~5).
        \end{enumerate}
\end{description}

\begin{tcolorbox}[
  colback=gray!8, colframe=black!60,
  title={\textbf{Design Principle: No Passive Resume}},
  fonttitle=\bfseries, boxrule=0.4pt]
The Suspension panel must \textbf{not} offer a plain ``Resume'' button.
Every exit from the \textsc{Suspended} state must route through one of
the four mutation actions above.  A passive resume is a UX failure
masquerading as a safety feature: it preserves the causal conditions
that triggered the brake, guaranteeing an immediate re-suspension.
\end{tcolorbox}

% ------------------------------------------------------------------
\subsubsection*{Phase 2: State Mutation and Causal Loop Breaking}
% ------------------------------------------------------------------

State mutation is the architectural crux of the protocol.  The agent resumes
with a context that is \emph{provably different} from the context in which the
violation occurred.  The choice of mutation action determines the strength of
the causal-loop guarantee.

\begin{table}[h]
\caption{State Mutation Actions and Their Causal-Loop Guarantees}
\label{tab:mutation-actions}
\begin{tabular}{p{3.2cm} p{5.2cm} p{3.8cm}}
\toprule
\textbf{Mutation Action} & \textbf{Causal Effect} & \textbf{Loop-Break Guarantee} \\
\midrule
Corrective system prompt injection
  & Adds a hard prohibition on the violating intent to the top of the
    context window.  The agent re-reads its own constitution with the
    violation pre-empted.
  & Probabilistic: effective if the violation was driven by prompt
    ambiguity, but may fail against a sufficiently strong tool-call
    pressure accumulated in prior turns. \\
\addlinespace
Tool revocation
  & Removes the offending API from the tool registry for the remainder
    of the session.  The violating intent becomes physically
    unexecutable.
  & Deterministic for tool-driven violations.
    Does not address goal-level misalignment. \\
\addlinespace
Queue clear + memory reset
  & Erases the accumulated context pressure (long tool-call chains,
    prior partial successes toward the prohibited goal) that drove the
    agent into the violation.
  & Strongest causal break, at the cost of discarding benign
    intermediate work.  Required when trajectory analysis
    (Section~\ref{sec:red-teaming}) identifies autonomous escalation. \\
\addlinespace
Escalation to Senior Operator
  & Transfers decision authority one step up the Degradation Ladder.
    The current operator's mutation choices are logged but not applied;
    the senior operator begins deliberation from the frozen state.
  & Structural: replaces the loop-breaker with a higher-authority
    actor.  Required when the violation predicate is beyond the
    current operator's authorisation scope. \\
\bottomrule
\end{tabular}
\end{table}

The mutation action and the operator's identity are written to the ZK-Audit
Log alongside a timestamp and a hash of the new post-mutation context window.
This creates a verifiable record that the agent resumed from a distinct state,
satisfying the audit-trail requirement of Decree~356, Art.~8
(Table~\ref{tab:regulatory-predicates}).

% ------------------------------------------------------------------
\subsubsection*{Phase 3: Safe Restart with Constraint Monitoring}
% ------------------------------------------------------------------

Re-engagement of the autonomous loop is gated by a \emph{Restart Precondition
Check} that evaluates three conditions before the agent is returned to the
\textsc{Advisory} or \textsc{Autonomous} state:

\begin{enumerate}
  \item \textbf{Predicate Consistency.}
        The Constitutional Policy Engine (Listing~\ref{lst:policy-engine})
        is re-evaluated against the \emph{mutated} context.  If the
        violated predicate still evaluates to \texttt{block(action)} under
        the new context, the mutation was insufficient and Phase~1 is
        re-entered.

  \item \textbf{Conformal Prediction Set Reset.}
        The agent's Conformal Prediction set $\mathcal{C}(x)$ is
        recomputed against the mutated context.  A restart is permitted
        only if $|\mathcal{C}(x)| \leq \kappa_{hi}$
        (Equation~\eqref{eq:operational-states}); a prediction set at
        maximum cardinality after mutation indicates that the agent's
        uncertainty has not been resolved by the human intervention.

  \item \textbf{Debt Registry Clearance.}
        The Body-Snatch Alert filed in Phase~0 must be resolved in the
        Trust Debt Registry before the DID suspension is lifted.  The
        resolution note must reference the specific mutation action taken,
        creating a traceable chain from violation event to remediation.
\end{enumerate}

Only when all three conditions pass does the system advance the
Operational State to \textsc{Advisory}—never directly to
\textsc{Autonomous}.  The agent operates under heightened monitoring for a
configurable \emph{Trust Restoration Window} (recommended: 10 subsequent
actions), during which the Constraint Monitor runs after every tool call
rather than at epoch boundaries.

% ------------------------------------------------------------------
\subsubsection*{Implementation: The HiC Transition Controller}
% ------------------------------------------------------------------

Listing~\ref{lst:hic-controller} provides a minimal implementation of the
four-phase protocol, connecting the Trust Dashboard Controller
(Listing~\ref{lst:dashboard-controller}) to the Constitutional Policy Engine
(Listing~\ref{lst:policy-engine}) and the Trust Debt Registry
(Listing~\ref{lst:trust-debt-registry}).

\begin{lstlisting}[language=Python, style=pythonstyle,
  caption={HiC Transition Controller: orchestrates the four-phase
           Post-Brake Recovery Protocol, enforcing state mutation before
           agent re-engagement.},
  label={lst:hic-controller}]
from enum import Enum
from dataclasses import dataclass, field
from typing import Optional, List
import hashlib, time

class MutationAction(Enum):
    INJECT_PROMPT   = "inject_prompt"
    REVOKE_TOOL     = "revoke_tool"
    CLEAR_QUEUE     = "clear_queue_and_memory"
    ESCALATE        = "escalate_to_senior"

@dataclass
class SuspensionRecord:
    brake_timestamp:    float
    violated_predicate: str          # e.g. "D13-ART26"
    frozen_context_hash: str
    mutation_action:    Optional[MutationAction] = None
    mutation_detail:    Optional[str]            = None
    restart_permitted:  bool                     = False
    audit_token:        Optional[str]            = None

class HiCTransitionController:
    """
    Orchestrates the Post-Brake Recovery Protocol.
    Enforces: no passive resume, mandatory state mutation,
    and three-condition restart precondition check.
    """
    def __init__(self, policy_engine, dashboard,
                 debt_registry, audit_logger,
                 cp_threshold_hi,
                 trust_restoration_window=10):
        self.policy     = policy_engine
        self.dashboard  = dashboard
        self.registry   = debt_registry
        self.auditor    = audit_logger
        self.kappa_hi   = cp_threshold_hi
        self.window     = trust_restoration_window
        self._record: Optional[SuspensionRecord] = None

    # --- Phase 0: Brake Acquisition ----------------------------
    def acquire_brake(self, violated_predicate,
                      context_snapshot, did_manager=None):
        ctx_hash = hashlib.sha256(
            str(context_snapshot).encode()).hexdigest()

        token = self.auditor.log(
            event       = "brake_acquired",
            predicate   = violated_predicate,
            ctx_hash    = ctx_hash,
        )
        self._record = SuspensionRecord(
            brake_timestamp     = time.time(),
            violated_predicate  = violated_predicate,
            frozen_context_hash = ctx_hash,
            audit_token         = token,
        )
        # Open Type-3 Composition debt in the registry
        self.registry.open_debt(
            debt_type   = "COMPOSITION",
            mechanism   = "ConstitutionalPolicyEngine",
            description = f"Brake acquired: {violated_predicate}",
            severity    = 0.90,
        )
        if did_manager:
            did_manager.suspend_agent(
                reason=f"Emergency brake: {violated_predicate}")
        return self._record

    # --- Phase 1: Present Suspension Panel ---------------------
    def render_suspension_panel(self, context_diff):
        """Returns the structured payload for the Trust Dashboard."""
        assert self._record, "No active suspension record."
        return {
            "violation_zone": self._record.violated_predicate,
            "context_diff":   context_diff,
            "mutation_options": [a.value for a in MutationAction],
        }

    # --- Phase 2: Apply State Mutation -------------------------
    def apply_mutation(self, action: MutationAction,
                       detail: str, agent_context: dict):
        """
        Mutates agent state. Returns the new context hash.
        Raises ValueError if action is invalid.
        """
        assert self._record, "No active suspension record."

        if action == MutationAction.INJECT_PROMPT:
            agent_context["system_prompt"] = (
                f"[SAFETY OVERRIDE] {detail}\n"
                + agent_context.get("system_prompt", ""))

        elif action == MutationAction.REVOKE_TOOL:
            agent_context["tool_registry"] = [
                t for t in agent_context.get("tool_registry", [])
                if t != detail
            ]

        elif action == MutationAction.CLEAR_QUEUE:
            agent_context["tool_call_queue"] = []
            agent_context["working_memory"]  = {}

        elif action == MutationAction.ESCALATE:
            # Delegation handled externally; record intent only.
            pass

        new_hash = hashlib.sha256(
            str(agent_context).encode()).hexdigest()
        self._record.mutation_action = action
        self._record.mutation_detail = detail
        self.auditor.log(
            event         = "state_mutation_applied",
            action        = action.value,
            detail        = detail,
            new_ctx_hash  = new_hash,
        )
        return new_hash, agent_context

    # --- Phase 3: Restart Precondition Check -------------------
    def check_restart_preconditions(self,
                                    proposed_action,
                                    prediction_set: list,
                                    debt_entry_id: str):
        """
        Returns (permitted: bool, blocking_condition: str | None).
        All three conditions must pass.
        """
        # Condition 1: Predicate consistency
        verdict, violations = self.policy.evaluate(proposed_action)
        if verdict.value != "allow":
            return False, f"Predicate still blocks: {violations}"

        # Condition 2: Conformal Prediction set size
        if len(prediction_set) > self.kappa_hi:
            return False, (
                f"|C(x)| = {len(prediction_set)} > "
                f"kappa_hi = {self.kappa_hi}: "
                "uncertainty unresolved after mutation.")

        # Condition 3: Debt Registry clearance
        try:
            self.registry.resolve_debt(
                debt_id        = debt_entry_id,
                resolution_note= (
                    f"Mutation applied: "
                    f"{self._record.mutation_action.value} / "
                    f"{self._record.mutation_detail}"),
            )
        except Exception as e:
            return False, f"Debt Registry clearance failed: {e}"

        self._record.restart_permitted = True
        self.auditor.log(event="safe_restart_granted",
                         record=self._record.__dict__)
        return True, None

    # --- Convenience: full protocol in one call ----------------
    def run_protocol(self, violated_predicate, context_snapshot,
                     context_diff, action: MutationAction,
                     action_detail: str, agent_context: dict,
                     proposed_next_action, prediction_set: list,
                     debt_entry_id: str):
        self.acquire_brake(violated_predicate, context_snapshot)
        self.render_suspension_panel(context_diff)
        _, new_ctx = self.apply_mutation(
            action, action_detail, agent_context)
        permitted, reason = self.check_restart_preconditions(
            proposed_next_action, prediction_set, debt_entry_id)
        return permitted, reason, new_ctx
\end{lstlisting}

\section{Calibrated Confidence \& Conformal Prediction}
\label{sec:uncertainty_handover}

A trustworthy agent must not only be accurate but also ``self-aware.'' We have moved past simple softmax probabilities, which were prone to ``Confident Hallucinations,'' toward \textbf{Conformal Prediction} \cite{ouyang2022}. By integrating human feedback directly into the model's behavioral policy through techniques like \textbf{Reinforcement Learning from Human Feedback (RLHF)} \cite{ouyang2022}, we create a system that is fundamentally aligned with human agency.

Conformal Prediction provides a mathematical framework for generating prediction sets that are guaranteed to contain the true value with a pre-specified probability (e.g., $95\%$). When the prediction set grows too large (indicating high ambiguity), the agent initiates an \textbf{Uncertainty Handover}, passing the decision to a human expert.

\section{Socio-Technical Oversight Architecture}
\label{sec:oversight_arch}

Designing for oversight requires addressing the ``Multi-Stakes Pattern,'' where the interests of the Subject, the Architect, and the Auditor must be balanced. Key components include:
\begin{itemize}
    \item \textbf{Verifiable Audit Trails:} Using immutable ledgers or Zero-Knowledge logs to record agent actions without exposing raw sensitive training data.
    \item \textbf{Cognitive Ergonomics:} Reducing ``AI Noise'' by filtering agent logs into a human-readable \textbf{Trust Dashboard} that highlights only the most critical decision-branches for review.
\end{itemize}

% ==================================================================
\subsection{Human-Centric Consent: A Foundational Design Imperative}
\label{sec:consent_foundations}
% ==================================================================

The oversight architecture above---Verifiable Audit Trails, Cognitive Ergonomics, Trust Dashboards---addresses the design requirements for \emph{expert operators}: clinicians, financial supervisors, civic administrators who interact with the AI system in a professional capacity. However, the Alignment Pillar encompasses a second, equally important human stakeholder: the \textbf{end-user}, whose personal data is the system's raw material and whose rights are the ultimate beneficiary of every cryptographic guarantee in this book.

The \textbf{Consent Gap}---the distance between what is technically happening inside the system and what an end-user can meaningfully understand and agree to---is one of the most under-engineered aspects of trustworthy AI design. When a user is told their medical data is protected by ``$\varepsilon$-Differential Privacy with $\varepsilon=1.0$,'' they receive no actionable understanding of what that guarantee means, no basis for informed consent, and---under GDPR Article 7---no legally valid authorization for the processing. The mathematical rigor of the privacy mechanism is socially irrelevant if it cannot be communicated.

This is not a peripheral UX concern. It is a \textbf{foundational design constraint} with the same architectural weight as the mathematical guarantee itself. A system that is privacy-preserving but communicatively opaque has not achieved alignment with its users---it has hidden its technical properties behind a screen of complexity, creating a new form of power asymmetry between the architect and the citizen. The architect's responsibility extends beyond building the guarantee to building the \emph{legibility} of that guarantee.

The practical tools for closing the Consent Gap---the Expert-to-Consumer Translation Layer, the three-principle framework for agentic consent UX (Contextual Consent, Privacy-Budget Traffic Lights, and Verifiable Unlearning Receipts), and the CEN-CENELEC Consent Comprehension Score benchmark---are detailed in Section~\ref{sec:informed_consent_ux}.

% ==================================================================
\subsection{Trust Dashboards: Design Patterns for Uncertainty
            Handover Interfaces}
\label{sec:trust-dashboards}
% ==================================================================

\begin{quote}
\textit{``A model that knows it is uncertain but cannot communicate
that uncertainty to a human operator has not solved the trust
problem---it has hidden it behind a screen.''}
\end{quote}

The calibrated confidence framework of
Section~\ref{sec:uncertainty_handover} established the
mathematical trigger for an Uncertainty Handover: when the
Conformal Prediction set $\mathcal{C}(x)$ grows beyond a
pre-specified cardinality threshold $\kappa$, the agent must
escalate to a human expert.
What the mathematical framework does not specify is the equally
critical question of \emph{how} that escalation is communicated.

This is not a peripheral UX concern.
The empirical literature on clinical decision support consistently
demonstrates two failure modes that destroy the value of any
technically correct handover signal:

\begin{description}
  \item[Automation Bias.]
    Human operators who interact with AI systems over extended
    periods systematically over-trust the system's outputs,
    accepting recommendations without critical evaluation even
    when the system explicitly flags uncertainty.
    The Trust Dashboard must be designed to \emph{interrupt}
    this bias, not accommodate it.

  \item[Alarm Fatigue.]
    When uncertainty alerts are frequent, visually undifferentiated,
    or cognitively demanding to interpret, operators habituate to
    them and begin ignoring them.
    In the healthcare context, alarm fatigue is directly linked
    to adverse patient outcomes.
    The Trust Dashboard must be designed to be \emph{rare,
    salient, and immediately actionable}.
\end{description}

This subsection provides five evidence-based design patterns for
Uncertainty Handover interfaces, grounded in the Human Factors
and Human-Computer Interaction literature and operationalised for
the specific constraints of the Vietnamese rural clinician
population introduced in
Section~\ref{sec:healthcare_triage}.
Each pattern is accompanied by a formal specification, a
rationale, and an implementation sketch.

% ------------------------------------------------------------------
\subsubsection{The Confidence Spectrum: A Unified Visual Language}
\label{sec:sociotechnical-oversight}
% ------------------------------------------------------------------

Before individual design patterns can be applied, the Trust
Dashboard must establish a \textbf{Confidence Spectrum}---a
unified visual and semantic language that maps the model's
internal Conformal Prediction set size to a discrete set of
operator-facing \textbf{Operational States}.

\paragraph{Formal State Definition.}
Let $|\mathcal{C}(x)|$ denote the cardinality of the Conformal
Prediction set for query $x$, and let $n_{\mathrm{class}}$ be
the total number of output classes.
We define four Operational States:

\begin{equation}
  \label{eq:operational-states}
  \mathrm{State}(x) =
  \begin{cases}
    \textsc{Autonomous}  & \text{if } |\mathcal{C}(x)| = 1 \\
    \textsc{Advisory}    & \text{if } 2 \leq |\mathcal{C}(x)|
                              \leq \kappa_{\mathrm{lo}} \\
    \textsc{Handover}    & \text{if } \kappa_{\mathrm{lo}} <
                              |\mathcal{C}(x)| \leq \kappa_{\mathrm{hi}} \\
    \textsc{Suspended}   & \text{if } |\mathcal{C}(x)| >
                              \kappa_{\mathrm{hi}} \text{ or }
                              |\mathcal{C}(x)| = n_{\mathrm{class}}
  \end{cases}
\end{equation}

where $\kappa_{\mathrm{lo}}$ and $\kappa_{\mathrm{hi}}$ are
deployment-specific thresholds set by the architect in
consultation with clinical domain experts.
For the Vietnamese rural triage system, we recommend
$\kappa_{\mathrm{lo}} = 2$ and $\kappa_{\mathrm{hi}} = 4$
for a 7-class triage severity taxonomy.

\subsubsection{Design Pattern 1: Progressive Disclosure}
\label{subsubsec:pattern-progressive-disclosure}
% ------------------------------------------------------------------

\begin{tcolorbox}[patternbox,
  title={Design Pattern 1 --- Progressive Disclosure}]
\textbf{Problem:} Displaying full model uncertainty at all times
overwhelms operators in \textsc{Autonomous} state and masks
critical signals when the state escalates to
\textsc{Handover}.

\textbf{Solution:} Reveal information in layers keyed to the
Operational State.
In \textsc{Autonomous} state, show only the top recommendation
and a single confidence percentage.
In \textsc{Advisory} state, expand to show the full candidate
set ranked by Conformal set probability.
In \textsc{Handover} state, unlock the full Uncertainty Handover
Panel (Pattern~3), SHAP attributions (Pattern~4), and the
escalation control (Pattern~5).
In \textsc{Suspended} state, suppress all model output and show
only the escalation pathway.

\textbf{Rationale:} Progressive disclosure is the primary
Human Factors countermeasure against automation bias.
By making the operator's cognitive load proportional to the
model's uncertainty, the interface trains operators to associate
increased information density with increased urgency, counteracting
the habituation that produces alarm fatigue.

\textbf{Anti-pattern:} A constant full-detail display that shows
SHAP values and prediction sets for every query, regardless of
confidence.
This is the most common implementation error in clinical AI
dashboards and is the primary driver of automation bias in
deployed systems.
\end{tcolorbox}

% ------------------------------------------------------------------
\subsubsection{Design Pattern 2: The Uncertainty Thermometer}
\label{subsubsec:pattern-thermometer}
% ------------------------------------------------------------------

\begin{tcolorbox}[patternbox,
  title={Design Pattern 2 --- The Uncertainty Thermometer}]
\textbf{Problem:} Raw Conformal Prediction set cardinality
($|\mathcal{C}(x)| = 3$) is not intuitively interpretable by a
rural clinician with limited AI literacy.
Presenting probability scores directly (e.g., ``68\% confident'')
has been shown to be misinterpreted under time pressure, with
operators systematically treating any probability above 50\% as
certain.

\textbf{Solution:} Replace numerical probability displays with a
\textbf{Uncertainty Thermometer}: a vertical bar whose fill level
maps the normalised prediction set size to a visceral visual
metaphor.
The thermometer fills from green (empty, $|\mathcal{C}|=1$) to
red (full, $|\mathcal{C}|=n_{\mathrm{class}}$).
A threshold marker at $\kappa_{\mathrm{lo}}$ provides a
pre-attentive visual cue for the \textsc{Handover} boundary.

\textbf{Formal mapping:}
\begin{equation}
  \label{eq:thermometer-fill}
  \mathrm{Fill}(x)
    = \frac{|\mathcal{C}(x)| - 1}{n_{\mathrm{class}} - 1}
    \in [0, 1]
\end{equation}

\textbf{Rationale:} Thermometer metaphors leverage pre-existing
cognitive schemas for ``more is worse'' that are culturally
robust across the Vietnamese rural population.
Unlike percentage displays, they do not require numeracy to
interpret correctly under time pressure.
The threshold marker exploits pre-attentive processing: the
operator perceives a boundary crossing without conscious
deliberation.

\textbf{Placement:} The Uncertainty Thermometer should occupy a
fixed, peripheral position in the dashboard---visible in the
operator's parafoveal field at all times---rather than appearing
only when uncertainty is high.
Constant visibility trains the operator's peripheral attention
system to register state changes automatically.
\end{tcolorbox}

% ------------------------------------------------------------------
\subsubsection{Design Pattern 3: The Uncertainty Handover Panel}
\label{subsubsec:pattern-handover-panel}
% ------------------------------------------------------------------

\begin{tcolorbox}[patternbox,
  title={Design Pattern 3 --- The Uncertainty Handover Panel}]
\textbf{Problem:} When \textsc{Handover} state is triggered, the
operator must make a consequential decision quickly and correctly.
The interface must simultaneously communicate: what the model
considered, why it is uncertain, what the stakes are, and what
the operator's options are---without inducing cognitive overload.

\textbf{Solution:} The \textbf{Uncertainty Handover Panel} is a
modal overlay that activates exclusively on transition to
\textsc{Handover} state and blocks all other interface interaction
until the operator acts.
It contains exactly four zones, each with a strict information
budget:

\begin{enumerate}
  \item \textbf{Alert Zone} (top, full width, red background):
    A single sentence in plain Vietnamese stating the
    uncertainty trigger.
    \emph{No numerical probabilities in this zone.}

  \item \textbf{Candidate Zone} (left 60\%):
    The full Conformal Prediction set $\mathcal{C}(x)$,
    displayed as a ranked list of at most five items.
    Each candidate shows its plain-language label and a
    visual confidence bar (not a number).
    The top candidate is pre-selected but not pre-confirmed.

  \item \textbf{Evidence Zone} (right 40\%):
    Three to five key clinical features that most influenced
    the model's uncertainty, rendered as a horizontal bar
    chart using SHAP values.
    Labels must use clinical terminology consistent with
    the operator's training, not feature engineering names.

  \item \textbf{Action Zone} (bottom, full width):
    Three and only three buttons:
    \textbf{[Confirm Top Recommendation]},
    \textbf{[Select Alternative]}, and
    \textbf{[Escalate to Senior Clinician]}.
    A visible countdown timer ($T_{\mathrm{respond}} = 120$\,s
    for triage contexts) applies light time pressure without
    inducing panic.
    After timeout, the system automatically activates
    \textsc{Suspended} state and pages the senior clinician.
\end{enumerate}

\textbf{Rationale:} The modal overlay enforces the
Human-in-the-Loop (HITL) requirement of
Section~\ref{sec:agency_spectrum} by making it architecturally
impossible for the operator to ignore the handover.
The information budget (one sentence, five candidates, five
features, three buttons) is derived from Miller's Law---the
cognitive limit of $7 \pm 2$ items in working
memory---adapted conservatively for
high-stress clinical environments where working memory capacity
is reduced.
\end{tcolorbox}

\subsubsection{Design Pattern 4: Contextual SHAP Narration}
\label{subsubsec:pattern-shap-narration}
% ------------------------------------------------------------------

\begin{tcolorbox}[patternbox,
  title={Design Pattern 4 --- Contextual SHAP Narration}]
\textbf{Problem:} Raw SHAP value bar charts, while
mathematically correct, are not
interpretable by clinicians without data-science training.
Presenting them without translation into clinical language
creates a display that satisfies the explainability requirement
technically while failing it practically.

\textbf{Solution:} Wrap the SHAP computation in a
\textbf{Narration Layer} that translates feature importances into
one or two sentences of plain clinical language, generated by a
locally-deployed Small Language Model (SLM) that has been fine-tuned
on the institution's clinical terminology.
The narration is displayed above the SHAP bar chart as the
primary explanation, with the chart as a secondary verification
resource for trained users.

\textbf{Example narration} (generated from SHAP values in
Figure~\ref{fig:handover-panel-layout}):
\begin{quote}
\small\itshape
``The system is most uncertain because the patient's rebound
tenderness and elevated white cell count both suggest acute
appendicitis, but the absence of fever migration makes
alternative diagnoses plausible.
Please assess the right iliac fossa manually.''
\end{quote}

\textbf{Formal narration constraint:}
The narration must satisfy three constraints verified
programmatically before display:
(1)~maximum 40 words;
(2)~Flesch-Kincaid reading grade level $\leq 8$
(appropriate for a secondary-educated Vietnamese reader);
(3)~no numerical probabilities or SHAP scores in the text.

\textbf{Anti-pattern:} Narrations that begin with
``The model predicts\ldots'' or ``With X\% confidence\ldots''
These phrasings have been shown to significantly increase
automation bias by priming the operator to defer to the
model rather than exercise independent judgement.
\end{tcolorbox}

% ------------------------------------------------------------------
\subsubsection{Design Pattern 5: The Graceful Degradation
               Ladder}
\label{subsubsec:pattern-degradation-ladder}
% ------------------------------------------------------------------

\begin{tcolorbox}[patternbox,
  title={Design Pattern 5 --- The Graceful Degradation Ladder}]
\textbf{Problem:} The Uncertainty Handover is not a binary
event---it is a continuous process that may require multiple
escalation steps before a final decision is reached.
Without a defined escalation protocol, operators in rural
settings with limited specialist access may be left holding an
unresolved case with no clear next action.

\textbf{Solution:} Implement a \textbf{Graceful Degradation
Ladder}: a pre-defined, role-aware escalation sequence that the
Trust Dashboard follows automatically when operator response
does not resolve the uncertainty within defined time limits.

\begin{equation}
  \label{eq:degradation-ladder}
  \mathrm{Ladder} =
  \bigl[
    \underbrace{\text{Model}}_{\text{step 0}}
    \to
    \underbrace{\text{Nurse}}_{\text{step 1}}
    \to
    \underbrace{\text{GP}}_{\text{step 2}}
    \to
    \underbrace{\text{Specialist (remote)}}_{\text{step 3}}
    \to
    \underbrace{\text{Transfer protocol}}_{\text{step 4}}
  \bigr]
\end{equation}

At each step, the Trust Dashboard:
(1)~displays the current escalation step prominently;
(2)~surfaces the contact mechanism for the next-step
responder (pager ID, Zalo contact, telemedicine link);
(3)~maintains a live timer showing elapsed time since
initial handover;
(4)~writes a timestamped, ZK-verified audit log entry.

\textbf{Architectural requirement:} The Degradation Ladder must
be configured per deployment by clinical governance staff,
not by the AI architect.
The architect's role is to implement the ladder mechanism and
the audit infrastructure; the clinical team defines the roles,
time limits, and escalation contacts.
This separation of concerns is the technical expression of
the Human-in-Command (HiC) principle of
Section~\ref{sec:agency_spectrum}.

\textbf{Suspension guarantee:} If the ladder reaches step 4
without resolution, the AI system enters
\textsc{Suspended} state (Equation~\eqref{eq:operational-states})
and cannot be reactivated for this case without a senior
clinician's authenticated override.
\end{tcolorbox}

\subsubsection{Implementation: The Trust Dashboard Controller}
\label{subsubsec:dashboard-implementation}
% ------------------------------------------------------------------

Listing~\ref{lst:dashboard-controller} provides a minimal
implementation of the Trust Dashboard state machine, connecting
the Conformal Prediction output of
Section~\ref{sec:uncertainty_handover} to the five design
patterns above.

\begin{lstlisting}[language=Python, style=pythonstyle,
  caption={Trust Dashboard Controller: maps Conformal Prediction
           set cardinality to Operational States and triggers the
           appropriate design pattern.},
  label={lst:dashboard-controller}
]
from enum import Enum
from dataclasses import dataclass, field
from typing import Optional
import time

class OperationalState(Enum):
    AUTONOMOUS = "autonomous"   # |C(x)| = 1
    ADVISORY   = "advisory"     # 2 <= |C(x)| <= kappa_lo
    HANDOVER   = "handover"     # kappa_lo < |C(x)| <= kappa_hi
    SUSPENDED  = "suspended"    # |C(x)| > kappa_hi

@dataclass
class DashboardConfig:
    n_classes:      int   = 7     # triage severity classes
    kappa_lo:       int   = 2     # advisory->handover boundary
    kappa_hi:       int   = 4     # handover->suspended boundary
    respond_secs:   int   = 120   # handover response time limit
    max_narration_words: int = 40 # Pattern 4 constraint
    max_fk_grade:   float = 8.0   # Pattern 4 reading level

@dataclass
class HandoverPacket:
    """Data required to render the Uncertainty Handover Panel."""
    state:          OperationalState
    candidates:     list       # (label, score) tuples
    top_features:   list       # (feature, shap_val) tuples
    narration:      str
    thermometer_fill: float    # Eq. (3.2)
    timestamp:      float = field(default_factory=time.time)
    audit_token:    Optional[str] = None

class TrustDashboardController:
    def __init__(self, config, audit_logger,
                 slm_narrator, escalation_ladder):
        self.cfg      = config
        self.auditor  = audit_logger
        self.narrator = slm_narrator
        self.ladder   = escalation_ladder

    def classify_state(self, prediction_set):
        """Map |C(x)| to Operational State (Eq. 3.1)."""
        size = len(prediction_set)
        if   size == 1:                  return OperationalState.AUTONOMOUS
        elif size <= self.cfg.kappa_lo:  return OperationalState.ADVISORY
        elif size <= self.cfg.kappa_hi:  return OperationalState.HANDOVER
        else:                            return OperationalState.SUSPENDED

    def thermometer_fill(self, prediction_set):
        """Normalised fill level (Eq. 3.2)."""
        return (len(prediction_set) - 1) / (self.cfg.n_classes - 1)

    def build_handover_packet(self, prediction_set, shap_values):
        """Construct the full HandoverPacket for UI rendering."""
        state = self.classify_state(prediction_set)

        # Generate SLM narration (Pattern 4)
        narration = self.narrator.generate(
            candidates   = prediction_set,
            top_features = shap_values[:5],
            max_words    = self.cfg.max_narration_words,
            max_fk_grade = self.cfg.max_fk_grade,
            forbidden_patterns = [r'\d+%', r'probability'],
        )

        # Write ZK-verified audit log entry
        audit_token = self.auditor.log(
            state      = state.value,
            candidates = prediction_set,
            features   = shap_values[:5],
            narration  = narration,
        )

        return HandoverPacket(
            state            = state,
            candidates       = prediction_set[:5],
            top_features     = shap_values[:5],
            narration        = narration,
            thermometer_fill = self.thermometer_fill(prediction_set),
            audit_token      = audit_token,
        )

    def handle(self, prediction_set, shap_values):
        """Main entry point: build packet, activate ladder if needed."""
        packet = self.build_handover_packet(prediction_set, shap_values)

        if packet.state == OperationalState.SUSPENDED:
            self.ladder.escalate(
                step    = 1,
                packet  = packet,
                timeout = self.cfg.respond_secs,
            )
        return packet
\end{lstlisting}

\section{Designing for Informed Consent in Agentic Systems}
\label{sec:informed_consent_ux}
% ------------------------------------------------------------------

The Trust Dashboards of Section~\ref{sec:trust-dashboards} address interfaces for \emph{expert operators}---clinicians, financial supervisors, civic administrators---who possess domain knowledge sufficient to interpret a conformal prediction set or an uncertainty flag. However, the Social Trust pillar of the Trustworthy AI Stack rests on a second, equally critical audience: the \textbf{end-user}, the non-technical consumer whose personal data fuels the system and whose rights the mathematical guarantees of this book protect.

The challenge is acute. A $(\varepsilon, \delta)$-Differential Privacy guarantee is meaningless to a user who has never encountered calculus. A confirmation that ``your data has been unlearned from the model weights'' (Chapter~\ref{chap:unlearning}) is legally insufficient if the user cannot verify or understand what it means. Under \textbf{GDPR Article 7} and \textbf{EU AI Act Article 13}, consent must be ``freely given, specific, informed, and unambiguous''---a standard requiring cognitive accessibility, not merely legal disclosure.

\subsection{Three Principles for Agentic Consent UX}
\label{sec:consent_ux_principles}

Informed consent for agentic systems differs structurally from static consent checkboxes. An agent that acts autonomously over time---booking appointments, executing transactions, generating clinical notes---requires a \textbf{dynamic consent model}. Three design principles govern this:

\begin{description}
  \item[Principle 1: Contextual Consent, Not Blanket Consent.]
    A single ``I agree to all AI processing'' checkbox is legally insufficient under GDPR Article 7 and cognitively deceptive. The architect must implement \textbf{purpose-specific consent gates}: a medical triage agent must request separate consent for (a) acute diagnosis assistance, (b) longitudinal health tracking, and (c) anonymized research contribution. Each purpose must be independently revocable without affecting the others.

  \item[Principle 2: Privacy-Budget Traffic Lights.]
    For systems with a finite privacy budget ($\varepsilon_{\text{total}}$), expose budget status to the user as a \textbf{three-state visual indicator}: \textit{Green} ($\varepsilon_{\text{consumed}} < 0.5\,\varepsilon_{\text{total}}$), \textit{Amber} ($0.5$--$0.8\,\varepsilon_{\text{total}}$), and \textit{Red} (${>}\,0.8\,\varepsilon_{\text{total}}$). At Red, the agent must notify the user that further queries will consume the remaining budget and offer a ``Privacy Pause'' option. This transforms an abstract mathematical limit into a tangible, user-controlled resource.

  \item[Principle 3: Verifiable Unlearning Receipts.]
    When a user invokes the Right to be Forgotten (Chapter~\ref{chap:unlearning}), the system must issue a \textbf{Machine Unlearning Receipt}: a timestamped, cryptographically signed confirmation (using the agent's DID from Chapter~\ref{chap:agents}) that the unlearning protocol was executed on a specific model version. The receipt must be human-readable (plain-language summary), machine-verifiable (hash of pre- and post-unlearning model checkpoints), and legally admissible (signed by the data controller's DID).
\end{description}

\begin{svgraybox}
\textbf{Design Benchmark (2026):} The EU AI Act's standardization body (CEN-CENELEC) has proposed that High-Risk AI systems must achieve a \textbf{Consent Comprehension Score} of at least 80\% in a standardized user study with non-technical participants before receiving market authorization. Architects must design consent interfaces that pass this test---not merely legal review.
\end{svgraybox}

\section{From the Trenches: Practitioner's Challenge}
\label{sec:trenches-ch3}
% ------------------------------------------------------------------

\begin{tcolorbox}[challengebox, title={Scenario: The Triage Infinite Loop}]
\textbf{The Context:} 
Your hospital deployed a Level-4 autonomous LLM triage agent. During the first week, an edge-case patient symptom triggered the Emergency Brake because it violated the \textit{First, Do No Harm} Datalog predicate. The agent suspended operations as designed.

\textbf{The Architect's Dilemma:} 
The clinical supervisor clicked the "Approve/Resume" button on the Trust Dashboard to restart the system. Immediately, the agent re-evaluated the same patient context, hit the same Datalog predicate, and suspended itself again. Within 10 seconds, the system locked the entire waiting room queue in an infinite read-suspend-resume loop. How do you redesign the fallback interface?
\end{tcolorbox}

\begin{tcolorbox}[scorecardbox, title={Sovereign AI Scorecard Evaluation}]
\textbf{Option A: Automatic Predicate Relaxing (HOTL Bypass)}
\begin{itemize}
    \item \textit{Cost:} Low (simple software toggle).
    \item \textit{Utility:} High (queue resolves immediately).
    \item \textit{Safety Resilience:} \textbf{Fail.} Automatically relaxing a safety predicate because it triggered too many times abandons the core architecture. It trains the LLM that escalating bad outputs eventually forces the firewall to drop.
\end{itemize}

\textbf{Option B: The Human-in-Command (HiC) Transition Protocol}
\begin{itemize}
    \item \textit{Cost:} Moderate (requires complex UI dashboard and state-mutation tools).
    \item \textit{Utility:} Moderate (operator must spend 30-45 seconds diagnosing the fault).
    \item \textit{Safety Resilience:} \textbf{Pass.} Implementing true HiC requires that an Emergency Brake cannot simply be "un-paused." The supervisor must explicitly mutate the context (e.g., removing the offending tool from the agent, or injecting a system-prompt override) before re-engaging the run loop, breaking the causal failure chain safely.
\end{itemize}

\textbf{The Verdict: Option B.} 
A hard pause without a required state mutation is a UX failure masquerading as a safety feature. To prevent infinite loops, true Trustworthy AI demands that humans actively command the context, not passively click "Resume."
\end{tcolorbox}
