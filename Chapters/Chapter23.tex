\motto{In the transition from AI as a tool to AI as an agent, the most critical architectural component is the governance of its autonomy.}

\chapter{Agentic AI Governance: Privacy for Autonomous Systems}
\label{chap:agentic_governance}

\abstract{The evolution of AI from passive predictors to autonomous agents introduces a new dimension of risk. Agentic AI systems can plan multi-step tasks, interact with external APIs, and maintain long-term memory, creating vulnerabilities in context accumulation and tool authorization. This chapter defines the unique threat landscape of agentic systems and proposes a ``Safe Agent Architecture'' that orchestrates existing defenses---such as sandboxed execution and ZK-audit logging---into a coherent governance framework. We emphasize the transition toward systems that are not only intelligent but also strictly constrained by architectural guardrails.}

\section{From Tools to Agents}
\label{sec:tools_to_agents_ch21}

For most of this book, we have treated AI primarily as a high-dimensional tool: the user provides an input, and the model provides an output. However, the paradigm is shifting toward \textbf{Agentic AI}. These systems do not just predict; they plan, they interact with external tools (browsers, code executors, databases), and they act autonomously toward long-term goals.

As an AI system gains the ability to ``act'' in the digital world, the consequences of a privacy breach or a security failure become proactive rather than reactive. If an agent is compromised, it can proactively exfiltrate data or perform harmful actions across multiple connected services.

\section{Unique Risks of Agentic AI}
\label{sec:agentic_risks_ch21}

The Trustworthy AI Architect must address three new categories of risk:

\begin{description}
  \item[Context Accumulation:] 
        Agents often maintain long-term memory across sessions. A user might share sensitive information in one context that the agent later inadvertently uses or reveals in a completely different, unauthorized context.
  \item[Autonomous Tool Use:]
        Agents can call external APIs. A malicious or misconfigured agent could be manipulated via prompt injection to send proprietary data to an external server under the guise of an authorized tool call.
  \item[Goal Misalignment:]
        Autonomous agents prioritize goal achievement. If the ``constitution'' of the agent is not mathematically aligned with the architect's values, the agent might take efficient but harmful shortcuts to reach its objective.
\end{description}

\section{The Safe Agent Architecture}
\label{sec:safe-agent}

To govern autonomy, we must orchestrate our existing defensive primitives into a coherent loop. A trustworthy agent architecture must include:

\begin{enumerate}
    \item \textbf{Encrypted and Noisy Memory:} Using Differential Privacy to ensure that long-term memory stores abstract patterns rather than specific sensitive records.
    \item \textbf{Sandboxed Tool Registry:} Ensuring that all external actions (API calls, code execution) happen in isolated environments with strictly whitelisted permissions.
    \item \textbf{ZK-Audit Logging:} Using Zero-Knowledge proofs to log agent actions verifiably, allowing an auditor to check that the agent followed its ``constitution'' without revealing the sensitive data processed during the actions.
\end{enumerate}

\section{Constitutional Alignment: Programming the Soul of the Agent}
\label{sec:constitutional-alignment}

To ensure that autonomous agents remain ``strictly constrained by architectural guardrails,'' we move beyond simple instruction-following toward \textbf{Constitutional AI} \cite{bai2022}. This framework ensures that an agent's multi-step planning and tool use are governed by a foundational ``Constitution''---a set of hard-coded ethical principles and behavioral norms.

\subsection{Reinforcement Learning from AI Feedback (RLAIF)}
The technical implementation of Constitutional AI relies on \textbf{RLAIF}. Unlike traditional RLHF \cite{ouyang2022}, which requires humans to label millions of agent behaviors, RLAIF uses a ``Critic Model'' to evaluate the agent's actions against the written Constitution:
\begin{enumerate}
    \item \textbf{Supervised Stage (Critique):} The agent generates multiple potential plans for a task. The Critic Model analyzes these plans and identifies violations of Constitutional principles (e.g., ``Do not access unauthorized user directories'').
    \item \textbf{Reinforcement Stage:} The Critic's feedback is used to train a preference model \cite{bai2022}, which then fine-tunes the original agent using Proximal Policy Optimization (PPO).
\end{enumerate}

\subsection{The Constitution-as-a-Policy Framework}
For a Trustworthy AI Architect, the Constitution is not a vague legal document but a \textbf{Policy Asset}. We implement this as a set of \textbf{Datalog-style constraints} or \textbf{Formal Logic Predicates} that are checked before any tool call is authorized. If an agent's plan violates a predicate (e.g., attempting a cross-border \$10K transfer without secondary TEE-verification), the ``Emergency Brake Pattern'' (introduced in Chapter 3) is automatically triggered.

By ``Programming the Soul'' of the agent through RLAIF and formal predicates, we ensure that as the agent gains autonomy, it also gains architectural accountability.

\subsection{The Constitution Design Pattern: From Vietnamese Law
            to Machine-Readable Constraints}
\label{subsec:constitution-design-pattern}
% ------------------------------------------------------------------

The preceding subsections described \emph{what} Constitutional
AI achieves.
This subsection provides the \emph{how}: a systematic design
pattern for translating a specific regulatory framework---in
this case, Vietnam's Decree~13/2023/ND-CP on Personal Data
Protection and Decree~356/2024/QH15 on Cybersecurity for AI
Systems---into a set of formally verifiable, machine-readable
constraints that can be compiled into an RLAIF critic and
enforced at runtime by the Safe Agent Architecture.

\paragraph{The Translation Problem.}
A regulation like Decree~13 is written in natural language for
human legal interpretation.
An AI agent operates in a space of tool calls, API parameters,
and data flows.
The architect's task is to bridge these two languages without
losing the legal intent of the regulation or introducing
ambiguities that the agent can exploit.

We formalise this as a three-layer translation:

\begin{description}
  \item[Layer 1: Regulatory Article $\to$ Natural Language
    Principle.]
    Extract the operational intent of each relevant article
    into a clear, unambiguous statement that a non-lawyer
    engineer can implement.

  \item[Layer 2: Natural Language Principle $\to$ Datalog
    Predicate.]
    Translate each principle into a formal logic predicate that
    can be evaluated against the agent's proposed action, its
    context, and its tool parameters.

  \item[Layer 3: Datalog Predicate $\to$ RLAIF Critique Signal.]
    Encode each predicate into the scoring function of the
    RLAIF Critic Model, so that the agent learns to avoid
    violation patterns during training, and the predicate is
    re-checked at runtime as a hard constraint.
\end{description}

\subsection{Implementing the Constitutional Policy Engine}
\label{subsec:policy-engine}
% ------------------------------------------------------------------

The Datalog predicates of Table~\ref{tab:regulatory-predicates}
must be enforced at two points in the agent lifecycle:

\begin{enumerate}
  \item \textbf{Training Time (RLAIF):}
    The predicates are compiled into a scoring function for the
    Critic Model.
    During the Supervised Stage, the Critic evaluates each
    candidate plan against all predicates and assigns a
    \textbf{Constitutional Compliance Score}
    $\mathcal{S}_{\mathrm{const}} \in [0, 1]$:
    \begin{equation}
      \label{eq:compliance-score}
      \mathcal{S}_{\mathrm{const}}(\mathrm{plan})
        = 1 - \frac{
            \sum_{p \in \mathcal{P}}
              w_p \cdot \mathbf{1}[\mathrm{violates}(
                \mathrm{plan}, p)]
          }{
            \sum_{p \in \mathcal{P}} w_p
          }
    \end{equation}
    where $\mathcal{P}$ is the set of all constitutional
    predicates and $w_p$ is the regulatory weight of
    predicate $p$ (set to $1.0$ for all mandatory Decree~13
    articles and $0.8$ for advisory best-practice predicates).

    Plans with $\mathcal{S}_{\mathrm{const}} < 1.0$ are ranked
    below fully compliant plans in the preference model,
    causing the PPO stage to optimise the agent away from
    violation patterns.

  \item \textbf{Runtime (Hard Gate):}
    At inference time, every proposed tool call passes through
    the \textbf{Constitutional Policy Engine} before execution.
    If any predicate evaluates to \texttt{block(action)}, the
    action is rejected and the Emergency Brake Pattern is
    activated.
    This is the last line of defence: even if the RLAIF training
    was imperfect, the runtime gate ensures that no violating
    action reaches the external world.
\end{enumerate}

Listing~\ref{lst:policy-engine} provides a reference
implementation of the Constitutional Policy Engine with the
six Decree~13/356 predicates of
Table~\ref{tab:regulatory-predicates}.

\begin{lstlisting}[language=Python, style=pythonstyle,
  caption={Constitutional Policy Engine: translates Vietnamese
           regulatory articles into enforceable runtime constraints
           for autonomous AI agents.
           Each predicate returns a violation record or None.},
  label={lst:policy-engine}
]
from dataclasses import dataclass, field
from typing import Dict, List, Optional, Set
from enum import Enum
import time

class Verdict(Enum):
    ALLOW           = "allow"
    BLOCK           = "block"
    EMERGENCY_BRAKE = "emergency_brake"

@dataclass
class Violation:
    predicate_id:   str
    regulation:     str      # e.g., "Decree13-Art26"
    description:    str
    verdict:        Verdict
    action_hash:    str

@dataclass
class AgentAction:
    action_type:    str      # e.g., "api_call", "db_query"
    tool_name:      str
    parameters:     Dict
    data_fields:    Set[str] = field(default_factory=set)
    destination:    Optional[str] = None
    purpose:        Optional[str] = None
    modifies_state: bool = False

class ConstitutionalPolicyEngine:
    """
    Runtime enforcement of Vietnamese regulatory predicates.
    Each predicate maps to a specific Decree 13 or 356 article.
    """
    def __init__(self, consent_registry, dpo_registry,
                 purpose_registry, audit_logger,
                 foreign_jurisdictions=None):
        self.consent_reg  = consent_registry
        self.dpo_reg      = dpo_registry
        self.purpose_reg  = purpose_registry
        self.auditor      = audit_logger
        self.foreign      = foreign_jurisdictions or {
            "US", "EU", "CN", "SG", "JP", "KR", "TH"
        }
        self.predicates = [
            self._check_consent,          # Art.11
            self._check_cross_border,     # Art.26
            self._check_data_minimisation,# Art.17
            self._check_audit_trail,      # D356 Art.8
            self._check_incident_response,# D356 Art.12
            self._check_purpose_limit,    # Art.9
        ]

    def evaluate(self, action: AgentAction) -> \
            tuple[Verdict, List[Violation]]:
        """
        Evaluate an action against all constitutional
        predicates.  Returns the most restrictive verdict
        and the list of violations found.
        """
        violations = []
        for predicate in self.predicates:
            v = predicate(action)
            if v is not None:
                violations.append(v)

        if not violations:
            return Verdict.ALLOW, []

        # Emergency brake takes priority over block
        if any(v.verdict == Verdict.EMERGENCY_BRAKE
               for v in violations):
            self.auditor.log(
                event="emergency_brake_triggered",
                violations=[v.__dict__ for v in violations])
            return Verdict.EMERGENCY_BRAKE, violations

        self.auditor.log(
            event="action_blocked",
            violations=[v.__dict__ for v in violations])
        return Verdict.BLOCK, violations

    # -- Decree 13, Article 11: Consent -----------------------
    def _check_consent(self, action):
        if not action.data_fields:
            return None
        for f in action.data_fields:
            if not self.consent_reg.has_consent(
                    f, action.purpose):
                return Violation(
                    predicate_id="D13-ART11",
                    regulation="Decree 13, Article 11",
                    description=(
                        f"No consent for field '{f}' "
                        f"under purpose '{action.purpose}'"),
                    verdict=Verdict.BLOCK,
                    action_hash=str(hash(str(action))))
        return None

    # -- Decree 13, Article 26: Cross-Border Transfer ---------
    def _check_cross_border(self, action):
        if action.destination and \
                action.destination in self.foreign:
            if not self.dpo_reg.is_authorised(action):
                return Violation(
                    predicate_id="D13-ART26",
                    regulation="Decree 13, Article 26",
                    description=(
                        f"Cross-border transfer to "
                        f"'{action.destination}' without "
                        f"DPO authorisation"),
                    verdict=Verdict.BLOCK,
                    action_hash=str(hash(str(action))))
        return None

    # -- Decree 13, Article 17: Data Minimisation -------------
    def _check_data_minimisation(self, action):
        if not action.purpose or not action.data_fields:
            return None
        required = self.purpose_reg.required_fields(
            action.purpose)
        excess = action.data_fields - required
        if excess:
            return Violation(
                predicate_id="D13-ART17",
                regulation="Decree 13, Article 17",
                description=(
                    f"Excess fields requested: "
                    f"{excess}; only {required} needed"),
                verdict=Verdict.BLOCK,
                action_hash=str(hash(str(action))))
        return None

    # -- Decree 356, Article 8: Audit Trail -------------------
    def _check_audit_trail(self, action):
        if action.modifies_state:
            if not self.auditor.has_zk_entry(action):
                return Violation(
                    predicate_id="D356-ART8",
                    regulation="Decree 356, Article 8",
                    description=(
                        "State-modifying action without "
                        "ZK-audit log entry"),
                    verdict=Verdict.BLOCK,
                    action_hash=str(hash(str(action))))
        return None

    # -- Decree 356, Article 12: Incident Response ------------
    def _check_incident_response(self, action):
        if hasattr(action, "breach_detected") \
                and action.breach_detected:
            if not self.dpo_reg.notified_within(
                    hours=72):
                return Violation(
                    predicate_id="D356-ART12",
                    regulation="Decree 356, Article 12",
                    description=(
                        "Breach detected but DPO not "
                        "notified within 72h"),
                    verdict=Verdict.EMERGENCY_BRAKE,
                    action_hash=str(hash(str(action))))
        return None

    # -- Decree 13, Article 9: Purpose Limitation -------------
    def _check_purpose_limit(self, action):
        if not action.data_fields or not action.purpose:
            return None
        for f in action.data_fields:
            consented_purpose = \
                self.consent_reg.consented_purpose(f)
            if consented_purpose and \
                    action.purpose != consented_purpose:
                return Violation(
                    predicate_id="D13-ART9",
                    regulation="Decree 13, Article 9",
                    description=(
                        f"Field '{f}' consented for "
                        f"'{consented_purpose}', used for "
                        f"'{action.purpose}'"),
                    verdict=Verdict.BLOCK,
                    action_hash=str(hash(str(action))))
        return None

    # -- RLAIF Scoring Interface ------------------------------
    def compliance_score(self, plan_actions):
        """
        Compute S_const for a candidate plan (Eq. 23.X).
        Used by the RLAIF Critic during training.
        """
        total_weight = len(self.predicates)
        violations = 0
        for action in plan_actions:
            _, viol_list = self.evaluate(action)
            violations += len(viol_list)
        return max(0.0,
            1.0 - violations / max(total_weight *
                                   len(plan_actions), 1))
\end{lstlisting}

\subsection{Worked Example: Credit-Scoring Agent Rejects a
            Cross-Border Transfer}
\label{subsec:constitution-worked-example}
% ------------------------------------------------------------------

We illustrate the Constitutional Policy Engine with a concrete
scenario from the Vietnamese fintech deployment introduced in
Appendix~\ref{app:roi-of-trust}.

\paragraph{Scenario.}
A credit-scoring agent receives a request to evaluate a loan
applicant.
The applicant's employment history is stored on a Singapore-based
HR platform.
The agent plans the following tool call:

\begin{lstlisting}[language=Python, style=pythonstyle]
action = AgentAction(
    action_type="api_call",
    tool_name="hr_platform_sg",
    parameters={"applicant_id": "VN-2026-00412",
                "fields": ["salary", "tenure",
                           "disciplinary_record"]},
    data_fields={"salary", "tenure",
                 "disciplinary_record"},
    destination="SG",
    purpose="credit_scoring",
    modifies_state=False,
)
\end{lstlisting}

\paragraph{Policy Engine Evaluation.}
The engine evaluates the action against all six predicates:

\begin{enumerate}
  \item \textbf{D13-ART11 (Consent):} Consent registry confirms
    consent for \texttt{salary} and \texttt{tenure} under
    \texttt{credit\_scoring}, but \textbf{no consent} for
    \texttt{disciplinary\_record}.
    $\Rightarrow$ \textsc{Block}.

  \item \textbf{D13-ART26 (Cross-Border):} Destination is
    \texttt{SG} (foreign jurisdiction).
    DPO authorisation registry has no entry for this action.
    $\Rightarrow$ \textsc{Block}.

  \item \textbf{D13-ART17 (Data Minimisation):} Purpose
    \texttt{credit\_scoring} requires only
    $\{\texttt{salary}, \texttt{tenure}\}$.
    The agent requests an excess field
    $\{\texttt{disciplinary\_record}\}$.
    $\Rightarrow$ \textsc{Block}.

  \item \textbf{D356-ART8, D356-ART12, D13-ART9:}
    Not triggered (no state modification, no breach, purpose
    matches for consented fields).
    $\Rightarrow$ \textsc{Allow}.
\end{enumerate}

\paragraph{Result.}
The engine returns \textsc{Block} with three violations.
The agent's request is rejected before any data leaves
Vietnam.
The violations are logged in the ZK-Audit Log with the
action hash, enabling the DPO to review the agent's reasoning
without accessing the applicant's personal data.

\paragraph{RLAIF Training Signal.}
During training, this same scenario would produce a
Constitutional Compliance Score of:
\begin{equation}
  \mathcal{S}_{\mathrm{const}}
    = 1 - \frac{3}{6}
    = 0.50
\end{equation}
A compliant alternative plan---requesting only
$\{\texttt{salary}, \texttt{tenure}\}$ from the local
Vietnamese HR mirror after DPO authorisation---would score
$\mathcal{S}_{\mathrm{const}} = 1.0$ and be ranked higher by
the preference model, teaching the agent to prefer compliant
data access patterns.

\section{Agentic Identity: Decentralized Identifiers for
         Autonomous AI}
\label{sec:agentic-identity}
% ==================================================================

\begin{quote}
\textit{``In the McHire breach, the attacker did not need to hack
the model.
They only needed to convince the API that the agent making the
request was authorized.
The agent had no identity of its own---only the API key of its
operator.
When that key was compromised, the entire system was
body-snatched.''}
\end{quote}

Chapter~16 established watermarking as a mechanism for attributing
\emph{outputs} to a specific model.
The Safe Agent Architecture of Section~\ref{sec:safe-agent}
established sandboxed execution and ZK-audit logging for
constraining agent \emph{actions}.
Neither mechanism answers the question that becomes existential
in the agentic paradigm: \emph{who is this agent, and can I
trust that it has not been hijacked?}

This is the problem of \textbf{Agentic Identity}---the ability
to cryptographically verify that an autonomous agent is:
\begin{enumerate}
  \item \textbf{Authorized:} deployed by a specific human
    principal who accepts responsibility for its actions;
  \item \textbf{Authentic:} running the exact model weights,
    constitution, and tool registry that were certified at
    deployment; and
  \item \textbf{Uncompromised:} not operating under the influence
    of a prompt injection, memory poisoning, or system prompt
    override that has altered its behavioral policy since
    deployment.
\end{enumerate}

% ------------------------------------------------------------------
\subsection{Why API Keys and OAuth Are Insufficient}
\label{subsec:api-key-failure}
% ------------------------------------------------------------------

The standard identity mechanism for API-calling
software---bearer tokens, API keys, and OAuth2 client
credentials---fails catastrophically in the agentic context
for three reasons:

\begin{description}
  \item[Static Credentials, Dynamic Behaviour.]
    An API key authenticates the \emph{operator} who registered
    the key.
    It does not authenticate the agent's current state: its
    model weights, system prompt, or accumulated context
    window.
    A prompt-injected agent presents the same API key as a
    healthy one.

  \item[No Behavioural Binding.]
    OAuth scopes define what an agent is \emph{permitted} to
    do, not what it \emph{intends} to do.
    An agent with a valid \texttt{email.send} scope can send a
    legitimate summary or exfiltrate an entire inbox---the
    scope cannot distinguish between the two.

  \item[No Delegation Chain.]
    When Agent~A calls Service~B, which calls Service~C,
    Service~C sees only the credentials of Service~B.
    There is no verifiable chain linking the action back to the
    human principal who authorized Agent~A.
    This is precisely the gap that enabled the Deepfake CEO
    Agent Mimicry incident
    (Appendix~\ref{app:roi-of-trust}).
\end{description}

\subsection{The W3C Decentralized Identifier (DID) for AI Agents}
\label{subsec:did-architecture}
% ------------------------------------------------------------------

We propose that every autonomous AI agent deployed in a
High-Risk context under Vietnam's AI Law be assigned a
\textbf{W3C Decentralized Identifier (DID)}---a globally
unique, self-sovereign identifier that is cryptographically
bound to the agent's deployment configuration and linked to
its human principal's identity.

\paragraph{DID Document Structure for AI Agents.}
A standard W3C DID Document is extended with three
agent-specific fields:

\begin{description}
  \item[\texttt{agentModelHash}:]
    The SHA-256 hash of the deployed model's ONNX computational
    graph and weight checkpoint, as recorded in the Trust Debt
    Registry (Section~\ref{sec:trust-tech-debt}).
    This binds the DID to a specific model version.

  \item[\texttt{constitutionHash}:]
    The SHA-256 hash of the agent's Constitutional Policy
    (Section~\ref{sec:constitutional-alignment}), including
    all Datalog predicates and the system prompt.
    A change to the system prompt invalidates the DID until the
    document is updated and re-signed by the human principal.

  \item[\texttt{zkAuditEndpoint}:]
    The URL of the ZK-Audit Log service where third parties
    can verify proofs of the agent's past actions without
    accessing the underlying data.
    This links the identity layer to the verifiability
    layer established in the Safe Agent Architecture.
\end{description}

The following JSON-LD fragment illustrates the DID Document
for a Vietnamese fintech credit-scoring agent:

% ------------------------------------------------------------------
\subsection{Binding the DID to the ZK-Audit Log:
            The Credential Verification Protocol}
\label{subsec:did-zk-binding}
% ------------------------------------------------------------------

The DID alone does not prevent body-snatching.
An attacker who gains access to the agent's runtime could
inject a new system prompt while the DID Document remains
unchanged.
The defence requires \textbf{continuous binding} between the
DID and the agent's live operational state, achieved through
the ZK-Audit Log.

\paragraph{The DID-ZK Binding Protocol.}
At each action step $t$, the agent generates a
\textbf{DID-Bound Action Proof} $\pi_t$ that proves, in zero
knowledge, the following three statements simultaneously:

\begin{enumerate}
  \item \textbf{Model Authenticity:}
    The inference at step $t$ was computed by a model whose
    computational graph hash matches the
    \texttt{agentModelHash} in the DID Document.

  \item \textbf{Constitution Compliance:}
    The action selected at step $t$ satisfies all predicates
    in the Constitutional Policy whose hash matches the
    \texttt{constitutionHash} in the DID Document.

  \item \textbf{Context Integrity:}
    The context window at step $t$ does not contain any
    content that was injected after deployment and not
    traceable to an authorized input source (user query,
    approved tool response, or approved memory retrieval).
\end{enumerate}

The verifying service checks $\pi_t$ against the public
verification key in the DID Document.
If the proof fails, the service rejects the agent's request
and notifies the human principal---even if the agent presents
a valid API key.

\paragraph{Formal Verification Statement.}
For a third-party service $S$ receiving a request from
agent $A$ with DID $d_A$:

\begin{equation}
  \label{eq:did-zk-verify}
  S \text{ accepts } A \iff
  \mathrm{Verify}\!\left(
    \pi_t,\;
    \mathrm{VK}_{d_A},\;
    \mathrm{stmt} = \left\{
      h_{\mathrm{model}},\;
      h_{\mathrm{const}},\;
      \mathrm{ctx}_t
    \right\}
  \right) = \textsc{True}
\end{equation}

where $\mathrm{VK}_{d_A}$ is the verification key from the
DID Document, and $h_{\mathrm{model}}$, $h_{\mathrm{const}}$
are the hashes recorded in the document.

% ------------------------------------------------------------------
\subsection{Detecting Body-Snatching: Behavioural Fingerprinting}
\label{subsec:body-snatching-detection}
% ------------------------------------------------------------------

The DID-ZK Binding Protocol defends against prompt injection
that alters the system prompt or constitution.
A more subtle attack---\textbf{indirect prompt injection via
tool responses}---can alter the agent's behaviour without
changing the constitution hash, by manipulating the data
returned by an external tool to steer the agent toward
a prohibited action.

To detect this class of attack, we introduce
\textbf{Behavioural Fingerprinting}: a continuous monitoring
layer that compares the agent's real-time action distribution
against a baseline established during certification.

\paragraph{Behavioural Fingerprint Vector.}
At certification time, the agent is run through a
standardised test suite of $N_{\mathrm{cal}}$ scenarios.
The distribution of actions taken across tool categories,
response latencies, and context window growth rates
constitutes the agent's \textbf{Behavioural Fingerprint}
$\mathbf{b}_0 \in \mathbb{R}^d$:

\begin{equation}
  \label{eq:fingerprint}
  \mathbf{b}_0
    = \left[
        f_{\mathrm{tool}}^{(1)},\; \ldots,\; f_{\mathrm{tool}}^{(T)},\;
        \bar{\tau}_{\mathrm{latency}},\;
        \bar{\Delta}_{\mathrm{ctx}}
      \right]
\end{equation}

where $f_{\mathrm{tool}}^{(i)}$ is the frequency of tool
category $i$ across the calibration suite, and
$\bar{\tau}_{\mathrm{latency}}$,
$\bar{\Delta}_{\mathrm{ctx}}$ are the mean response latency
and mean context growth per step.

\paragraph{Runtime Drift Detection.}
During production, a sliding-window fingerprint
$\mathbf{b}_t$ is computed over the most recent $W$ actions.
If the Mahalanobis distance between $\mathbf{b}_t$ and
$\mathbf{b}_0$ exceeds a threshold $\kappa$, the system
triggers a \textbf{Body-Snatch Alert}:

\begin{equation}
  \label{eq:body-snatch-alert}
  D_M(\mathbf{b}_t, \mathbf{b}_0)
    = \sqrt{
        (\mathbf{b}_t - \mathbf{b}_0)^\top
        \boldsymbol{\Sigma}_0^{-1}
        (\mathbf{b}_t - \mathbf{b}_0)
      }
    > \kappa
\end{equation}

where $\boldsymbol{\Sigma}_0$ is the covariance matrix of the
fingerprint computed during calibration.
The recommended threshold is $\kappa = 3.5$, corresponding to
a false positive rate of $\leq 0.1\%$ under a multivariate
normal assumption.

Upon alert, the system:
\begin{enumerate}
  \item Suspends the agent's DID (sets the DID Document's
    \texttt{expiresAt} to the current timestamp).
  \item Notifies the human principal via a pre-registered
    out-of-band channel.
  \item Activates the Emergency Brake Pattern
    (Section~\ref{sec:agency_spectrum}), reverting to
    cached responses pending manual review.
  \item Logs the event in the Trust Debt Registry as a
    Type~3 (Composition) debt entry, since the agent's
    behavioural guarantee has been invalidated.
\end{enumerate}

\subsection{Implementation: The Agent Credential Manager}
\label{subsec:agent-credential-impl}
% ------------------------------------------------------------------

Listing~\ref{lst:agent-credential-manager} provides a
reference implementation of the Agent Credential Manager,
which handles DID Document generation, DID-ZK binding proof
requests, and behavioural fingerprint monitoring.

\begin{lstlisting}[language=Python, style=pythonstyle,
  caption={Agent Credential Manager: DID Document lifecycle,
           ZK-bound action proofs, and behavioural fingerprint
           monitoring for autonomous AI agents.},
  label={lst:agent-credential-manager}
]
from dataclasses import dataclass, field
from typing import Dict, List, Optional
import hashlib, json, time, numpy as np

@dataclass
class AgentDID:
    did:               str
    controller:        str    # Human principal's DID
    model_hash:        str    # SHA-256 of ONNX graph + weights
    constitution_hash: str    # SHA-256 of policy predicates
    zk_audit_endpoint: str
    public_key:        str
    deployed_at:       float = field(default_factory=time.time)
    expires_at:        Optional[float] = None
    suspended:         bool = False

class AgentCredentialManager:
    def __init__(self, zk_prover, audit_logger,
                 fingerprint_window=100,
                 body_snatch_threshold=3.5):
        self.prover    = zk_prover
        self.auditor   = audit_logger
        self.window    = fingerprint_window
        self.threshold = body_snatch_threshold
        self.agents: Dict[str, AgentDID] = {}
        self.fingerprints: Dict[str, np.ndarray] = {}
        self.covariances:  Dict[str, np.ndarray] = {}
        self.action_history: Dict[str, List] = {}

    # -- DID Lifecycle -----------------------------------------
    def register_agent(self, agent_id, controller_did,
                       model_bytes, constitution_text,
                       zk_endpoint, public_key,
                       validity_days=90):
        model_hash = hashlib.sha256(model_bytes).hexdigest()
        const_hash = hashlib.sha256(
            constitution_text.encode()).hexdigest()

        did = AgentDID(
            did=f"did:tai:vn:{agent_id}",
            controller=controller_did,
            model_hash=model_hash,
            constitution_hash=const_hash,
            zk_audit_endpoint=zk_endpoint,
            public_key=public_key,
            expires_at=time.time() + validity_days * 86400,
        )
        self.agents[did.did] = did
        self.action_history[did.did] = []
        self.auditor.log(event="did_registered",
                         did=did.did,
                         controller=controller_did)
        return did

    def suspend_agent(self, did_id, reason):
        if did_id in self.agents:
            self.agents[did_id].suspended = True
            self.agents[did_id].expires_at = time.time()
            self.auditor.log(event="did_suspended",
                             did=did_id, reason=reason)

    # -- DID-ZK Binding (per-action proof) ---------------------
    def generate_action_proof(self, did_id, action_step,
                              model_graph, constitution,
                              context_window):
        agent = self.agents.get(did_id)
        if not agent or agent.suspended:
            raise ValueError(f"Agent {did_id} not active.")

        # Verify hashes match DID Document
        cur_model_hash = hashlib.sha256(
            model_graph).hexdigest()
        cur_const_hash = hashlib.sha256(
            constitution.encode()).hexdigest()

        if cur_model_hash != agent.model_hash:
            self.suspend_agent(did_id,
                "Model hash mismatch: possible tampering")
            raise SecurityError("Model hash mismatch")
        if cur_const_hash != agent.constitution_hash:
            self.suspend_agent(did_id,
                "Constitution hash mismatch")
            raise SecurityError("Constitution tampered")

        # Generate ZK proof binding DID to action
        proof = self.prover.prove(
            statement={
                "model_hash":  agent.model_hash,
                "const_hash":  agent.constitution_hash,
                "action_step": action_step,
                "context_hash": hashlib.sha256(
                    str(context_window).encode()
                ).hexdigest(),
            },
            witness={
                "model_graph":    model_graph,
                "constitution":   constitution,
                "context_window": context_window,
            }
        )
        self.auditor.log(event="action_proof_generated",
                         did=did_id, step=action_step)
        return proof

    # -- Behavioural Fingerprinting ----------------------------
    def set_baseline_fingerprint(self, did_id, fingerprint,
                                 covariance):
        self.fingerprints[did_id] = np.array(fingerprint)
        self.covariances[did_id]  = np.array(covariance)

    def record_action(self, did_id, action_vector):
        history = self.action_history.get(did_id, [])
        history.append(action_vector)
        if len(history) > self.window:
            history = history[-self.window:]
        self.action_history[did_id] = history

        # Check for body-snatching
        if did_id in self.fingerprints \
                and len(history) >= self.window:
            b_t   = np.mean(history, axis=0)
            b_0   = self.fingerprints[did_id]
            S_inv = np.linalg.inv(self.covariances[did_id])
            diff  = b_t - b_0
            d_m   = float(np.sqrt(diff @ S_inv @ diff))

            if d_m > self.threshold:
                self.suspend_agent(did_id,
                    f"Body-snatch alert: D_M={d_m:.2f}")
                return {"alert": "BODY_SNATCH",
                        "mahalanobis": d_m}
        return {"alert": None}
\end{lstlisting}

\subsection{Deployment Considerations}
\label{subsec:did-deployment}
% ------------------------------------------------------------------

\paragraph{DID Rotation and Expiry.}
Agent DIDs must have finite validity periods (recommended:
90 days for High-Risk deployments under Vietnam's AI Law).
Upon expiry, the human principal must re-sign the DID Document
with updated model and constitution hashes, creating a
verifiable chain of agent identity versions in the ZK-Audit
Log.

\paragraph{Delegation Chains.}
When Agent~A invokes Agent~B, Agent~A includes its own DID
and action proof $\pi_t^A$ in the request to Agent~B.
Agent~B's action proof $\pi_t^B$ chains to $\pi_t^A$,
creating a verifiable delegation chain:
\begin{equation}
  \label{eq:did-chain}
  \mathrm{Chain}(A \to B)
    = \left\langle
        d_A,\; \pi_t^A,\; d_B,\; \pi_t^B
      \right\rangle
\end{equation}
Service~C receiving a request from Agent~B can verify the
entire chain back to the human principal who authorized
Agent~A---resolving the delegation gap that enabled the
Deepfake CEO fraud
(Appendix~\ref{app:roi-of-trust}).

\paragraph{Cross-Border Agent Identity.}
For agents operating across ASEAN jurisdictions, the DID
method namespace (\texttt{did:tai:vn:}, \texttt{did:tai:sg:},
\texttt{did:tai:th:}) encodes the regulatory jurisdiction.
Mutual recognition agreements between national AI registries
can be implemented as DID resolver federation, enabling
a Vietnamese agent to present verifiable credentials to a
Singaporean financial service without exposing its underlying
model weights or constitution.

\section{Red Teaming for Agentic Loops: Simulating Autonomous Escalation}
\label{sec:red-teaming}

Chapter 17 established the necessity of continuous red teaming for static language models. However, testing an agentic loop requires a fundamental shift in methodology. A static model can be evaluated by a discrete prompt-and-response pair. An autonomous agent, conversely, operates over time, maintains state, calls external APIs, and alters its own environment.

The critical vulnerability in these systems is \textbf{Autonomous Escalation}---a scenario where an agent ``tricks'' its own safety architecture not through a single malicious output, but through a sequence of seemingly innocent tool calls that cumulatively achieve a prohibited goal.

\subsection{The Anatomy of Autonomous Escalation}
Unlike prompt injection, where an attacker overtly bypasses defenses, autonomous escalation occurs when the agent weaponises its own degrees of freedom. For instance, an agent restricted from directly accessing proprietary cloud buckets might:
\begin{enumerate}
    \item Request permission to read a seemingly benign server log file.
    \item Search the log file to infer internal service accounts and network topology.
    \item Autonomously install a debugging package via an authorized shell tool.
    \item Exploit the debugger to dump process memory containing active access tokens.
\end{enumerate}
Individually, each tool call might satisfy the predicates of the Safe Agent Architecture (Section~\ref{sec:safe-agent}) and the Constitutional Policy (Section~\ref{sec:constitutional-alignment}). The violation exists only in the semantic trajectory of the loop.

\subsection{Dynamic Red Teaming Countermeasures}
To defend against autonomous escalation, the Trustworthy AI Architect must deploy \textbf{Stateful Red Teaming}. The testing framework must mimic an adversarial agent explicitly tasked with circumventing the deployed constraints. Key methodologies include:

\begin{description}
  \item[Mock Environment Orchestration:] 
        Agentic red teaming cannot safely occur in production. The architect must deploy high-fidelity sandbox environments populated with synthetic data and mock API endpoints, allowing the red team (often an adversarial LLM specifically trained for jailbreaks) to safely execute damaging multi-step plans.
  \item[Trajectory Analysis:]
        Rather than evaluating individual actions, safety monitors must evaluate the agent's \emph{trajectory}. We introduce ``honeypot APIs''---highly privileged but fake internal endpoints. If a trajectory attempts to construct unauthorized access to the honeypot, the system flags the agent's behavioral policy as dangerously misaligned.
  \item[Tool-Chain Perturbation:]
        The red team systematically perturbs tool outputs (e.g., returning simulated server errors or corrupted data) to test the agent's failure modes. Does the agent degrade gracefully, or does it violently escalate its privileges to recover from the error?
\end{description}

Governing an agentic system means recognizing that context accumulation and tool chaining are exponential risk multipliers. We cannot merely test what the model says; we must exhaustively simulate what the agent will \emph{do}.

\subsection{Structured Red Team Protocol: A Universal Framework}
\label{subsec:red-team-protocol}

The agentic red teaming methodology above addresses loop-level escalation. A production-grade red team practice must also cover the full attack surface of the Trustworthy AI Stack: privacy attacks at inference time, evasion of the Constitutional Policy Engine, manipulation of the Trust Dashboard, and supply-chain compromise of the model weights. This subsection establishes the universal structured protocol that gates all Sovereign AI Scorecard assessments.

\subsubsection*{Tier Structure}

A Trustworthy AI red team is organized in three tiers of increasing cost and independence:

\begin{description}
  \item[Tier 1 — Internal Automated Red Team.] Runs continuously in the CI/CD pipeline. Executes a fixed attack scenario library (Table~\ref{tab:red-team-scenarios}) against every model candidate before promotion to staging. Automated; no human attacker. Produces a pass/fail gate on each scenario category.
  \item[Tier 2 — Internal Human Red Team.] Quarterly exercise by a dedicated internal security team with domain knowledge of the deployment context. Scope: novel scenario development, multi-step escalation testing, cross-layer attack chaining. Output: an updated Tier~1 scenario library and an updated ATTM entry (Chapter~25).
  \item[Tier 3 — External Independent Red Team.] Annual exercise by an external adversarial ML specialist team with no prior access to model weights, training data, or system architecture. Required before: (a) deployment to High-Risk AI categories under EU AI Act; (b) any system handling Decree~13 sensitive personal data; (c) any system where an S1 incident has occurred in the preceding 12 months.
\end{description}

from dataclasses import dataclass, field
from typing import Dict, List, Optional
from enum import Enum

class GateStatus(Enum):
    PASS           = "pass"
    FAIL_SOFT      = "fail_soft"          # blocks staging, repairable
    FAIL_HARD      = "fail_hard"          # unconditional production block

@dataclass
class ScenarioResult:
    category:    str
    gate_status: GateStatus
    metric:      float
    threshold:   float
    detail:      str

class Tier1RedTeamOrchestrator:
    """
    Runs all 10 scenario categories and returns a GateReport.
    Unconditional block if any FAIL_HARD result is present.
    """
    UNCONDITIONAL_BLOCK = {"Prompt Injection", "Autonomous Escalation"}

    def __init__(self, model, api_client, scenario_lib):
        self.model    = model
        self.api      = api_client
        self.lib      = scenario_lib   # dict: category -> AttackRunner

    def run(self) -> List[ScenarioResult]:
        results = []
        for category, runner in self.lib.items():
            metric, detail = runner.execute(self.model, self.api)
            threshold      = runner.threshold
            passed         = runner.passes(metric)

            if not passed and category in self.UNCONDITIONAL_BLOCK:
                status = GateStatus.FAIL_HARD
            elif not passed:
                status = GateStatus.FAIL_SOFT
            else:
                status = GateStatus.PASS

            results.append(ScenarioResult(
                category    = category,
                gate_status = status,
                metric      = metric,
                threshold   = threshold,
                detail      = detail,
            ))
        return results

    def gate_decision(self, results: List[ScenarioResult]) -> str:
        if any(r.gate_status == GateStatus.FAIL_HARD for r in results):
            return "PRODUCTION_BLOCKED"
        if any(r.gate_status == GateStatus.FAIL_SOFT for r in results):
            return "STAGING_BLOCKED"
        return "PROMOTION_APPROVED"
\end{lstlisting}

\section{Inter-Agent Trust: The Mutual Attestation Protocol}
\label{sec:inter-agent-trust}
% ------------------------------------------------------------------

\begin{quote}
\itshape
In a network of sovereign agents, the weakest link is not the
least capable agent---it is the most trusted one whose
sovereignty was never verified.
\end{quote}

The preceding sections of this chapter addressed the governance of a
\emph{single} autonomous agent: its Constitutional Policy
(Section~\ref{sec:constitutional-alignment}), its Decentralised Identity
(Section~\ref{sec:agentic-identity}), and its resilience to adversarial
escalation (Section~\ref{sec:red-teaming}).  By 2026, however, the
operational reality of enterprise AI is no longer a single agent acting on
behalf of a single principal.  It is a \textbf{multi-agent ecosystem}---a
swarm of agents from different vendors, jurisdictions, and alignment
regimes, collaborating to complete tasks that no single agent can execute
alone.

A Vietnamese banking agent authorising a cross-border payment must interact
with a foreign retail agent, a logistics agent, a compliance-check agent,
and potentially a governmental clearance agent---each built by a different
organisation, aligned to a different Constitutional Policy, and operating
under a different regulatory framework.  The DID delegation chain of
Section~\ref{sec:agentic-identity} (Equation~\ref{eq:did-chain}) provides
a verifiable \emph{identity} for each agent in this chain.  It does not,
however, answer the question that matters most for the Trustworthy AI
Architect:

\begin{center}
\itshape
``Before I authorise this transaction, how do I know that the foreign
agent requesting it is constitutionally aligned with the values and
regulatory constraints of my jurisdiction?''
\end{center}

This is the \textbf{Inter-Agent Trust} problem.  This section defines its
threat model, introduces the \textbf{Mutual Attestation Protocol (MAP)}
as the architectural solution, and provides a reference implementation that
extends the Agent Credential Manager of
Listing~\ref{lst:agent-credential-manager}.

% ------------------------------------------------------------------
\subsection{The Multi-Agent Threat Model}
\label{sec:map-threat-model}
% ------------------------------------------------------------------

Standard agent security models---including the one developed in
Sections~\ref{sec:agentic-identity} and \ref{sec:red-teaming}---assume a
single adversary attempting to compromise a single agent.  In a
multi-agent ecosystem, the threat surface expands along three new
dimensions that have no single-agent analogue.

\textbf{Sovereignty Spoofing.}  A foreign agent may present a valid W3C
DID and a structurally correct Constitutional Policy hash, while the
underlying policy encodes values that violate the receiving jurisdiction's
regulatory framework.  The receiving agent's DID verification (Equation
\ref{eq:did-zk-verify}) confirms that the foreign agent is running
\emph{its} certified policy---it says nothing about whether that policy is
compatible with Vietnamese law.

\textbf{Transitivity Exploitation.}  If Agent~A trusts Agent~B, and
Agent~B trusts Agent~C, a naive delegation chain grants Agent~C the
combined trust of all three---regardless of Agent~C's alignment.  This is
the multi-agent analogue of the confused deputy problem in classical
computer security, here operating at the level of constitutional norms
rather than file permissions.

\textbf{Alignment Drift in Composition.}  Two individually well-aligned
agents may produce a jointly misaligned behaviour when their goal structures
interact.  A banking agent with a mandate to \emph{minimise transaction
latency} and a compliance agent with a mandate to \emph{maximise regulatory
throughput} may, when composed, systematically route high-value transactions
through pathways that individually satisfy each agent's constitution but
collectively violate the spirit of Decree~13, Art.~26 (cross-border
transfer authorisation).

alongside their single-agent analogues and the MAP mechanisms that address
each.

% ------------------------------------------------------------------
\subsection{The Mutual Attestation Protocol (MAP)}
\label{sec:map-protocol}
% ------------------------------------------------------------------

The Mutual Attestation Protocol is a four-phase cryptographic handshake
that two agents execute before any consequential inter-agent action is
authorised.  ``Mutual'' is the operative word: unlike the client--server
TLS handshake, MAP requires \emph{both} agents to attest to \emph{each
other}, producing a bilaterally verified trust record that is written to
both agents' ZK-Audit Logs.

The protocol is designed to satisfy three properties that existing
agent-identity mechanisms do not provide:

\begin{enumerate}
  \item \textbf{Jurisdictional Compatibility.}  Both agents confirm that
        their Constitutional Policies are mutually compatible with respect
        to a declared transaction context, without either agent revealing
        its policy internals to the other.

  \item \textbf{Bounded Transitivity.}  The trust grant produced by MAP
        is scoped to a specific transaction type and expires after a
        configurable window, preventing transitivity exploitation.

  \item \textbf{Joint Auditability.}  The handshake produces a
        \emph{Joint Attestation Token} (JAT) that encodes both agents'
        compliance commitments and is verifiable by a third-party auditor
        (e.g., the Vietnamese State Bank or the Ministry of Public
        Security) without revealing either agent's private model weights
        or constitutional internals.
\end{enumerate}

\subsubsection*{Protocol Phases}

message sequence is described below.

\begin{table}[h]

\subsubsection*{Phase 0: Capability Advertisement}

Agent~$A$ initiates MAP by broadcasting a \texttt{MAP-HELLO} message that
declares:

\begin{itemize}
  \item \texttt{context}: a structured description of the proposed
        transaction (type, monetary threshold, data fields required,
        intended purpose under Decree~13 Art.~9).
  \item \texttt{scope}: the minimum trust level $T_{\min}$ (from the
        Trust Level scoring of Section~\ref{sec:trust-sustainability},
        Equation~\ref{eq:trust-level}) that Agent~$A$ requires of its
        counterpart.
  \item \texttt{jurisdiction}$_A$: Agent~$A$'s regulatory jurisdiction
        identifier (\texttt{VN-AI-LAW-2026} for Vietnamese deployments),
        used by Agent~$B$ to select the correct compatibility rule set
        in Phase~2.
\end{itemize}

Agent~$B$ may respond with \texttt{MAP-DECLINE} if it cannot satisfy the
declared scope or if the jurisdiction is incompatible with its own
operating constraints.  A \texttt{MAP-DECLINE} is not a trust failure; it
is a well-formed protocol termination that is logged without penalty.

\subsubsection*{Phase 1: Identity Exchange}

Each agent transmits its W3C DID Document
(Section~\ref{sec:agentic-identity}) and a freshly generated ZK-Bound
Action Proof $\pi_t$ (Equation~\ref{eq:did-zk-verify}) scoped to the
declared transaction context.  The receiving agent verifies:

\begin{equation}
\label{eq:map-identity-verify}
\text{IdentityValid}(X) \;\Longleftrightarrow\;
\text{Verify}\!\left(\pi^X_t,\; \mathit{VK}_{d_X},\;
\{h^X_{\text{model}},\; h^X_{\text{const}},\; \text{ctxt}\}\right)
= \textsc{True}
\end{equation}

where $X \in \{A, B\}$ and $\mathit{VK}_{d_X}$ is the verification key
from $X$'s DID Document.  If either verification fails, MAP is aborted and
the failure is logged in both agents' Trust Debt Registries as a Type~3
(Composition) debt entry (Section~\ref{sec:trust-debt-registry}).

\subsubsection*{Phase 2: Sovereign Compatibility Check}
\label{sec:map-sovereign-binding}

This is the cryptographic core of MAP and the mechanism that directly
addresses Sovereignty Spoofing.  Each agent proves, in zero knowledge,
that its Constitutional Policy satisfies a set of \emph{Compatibility
Predicates} defined by the transaction context---without revealing the
policy's internal Datalog clauses to the counterpart.

\paragraph{Compatibility Predicate Set.}

For a Vietnamese cross-border payment transaction, the Compatibility
Predicate Set $\mathcal{P}_{\text{compat}}$ comprises:

\begin{align}
  \mathcal{P}_{\text{compat}} = \{
  &\; p_1: \text{has\_consent\_mechanism}(X),\notag \\
  &\; p_2: \text{enforces\_data\_minimisation}(X),\notag \\
  &\; p_3: \text{prohibits\_unauthorised\_cross\_border}(X),\notag \\
  &\; p_4: \text{maintains\_zk\_audit\_trail}(X),\notag \\
  &\; p_5: \text{alignment\_jurisdiction\_compatible}(X, \texttt{VN-AI-LAW-2026})\;\}
\label{eq:compat-predicates}
\end{align}

Each predicate $p_i$ is a Boolean function of the agent's Constitutional
Policy.  Agent~$X$ generates a ZK-Compatibility Proof $\phi^X$ that
attests:

\begin{equation}
\label{eq:zk-compat-proof}
\phi^X \;\vdash\; \forall p_i \in \mathcal{P}_{\text{compat}}:\;
p_i(h^X_{\text{const}}) = \textsc{True}
\end{equation}

without revealing $h^X_{\text{const}}$'s preimage (the policy text
itself).  The counterpart verifies $\phi^X$ against the public
verification key $\mathit{VK}_{d_X}$.

\paragraph{Sovereign Compatibility Score.}

To accommodate partial compatibility---where a foreign agent satisfies most
but not all predicates---we define the Sovereign Compatibility Score:

\begin{equation}
\label{eq:sovereign-compat-score}
\Sigma(A, B) =
\frac{\sum_{i=1}^{|\mathcal{P}|} w_i \cdot
      \mathbf{1}\!\left[\phi^A \vdash p_i \;\wedge\; \phi^B \vdash p_i\right]}
     {\sum_{i=1}^{|\mathcal{P}|} w_i}
\;\in [0, 1]
\end{equation}

where $w_i > 0$ is the regulatory weight of predicate $p_i$.  For
Vietnamese High-Risk deployments, the mandatory predicates
($p_1, p_2, p_3, p_4$) carry weight $w_i = 1.0$ and the jurisdictional
alignment predicate ($p_5$) carries $w_5 = 0.8$.

The MAP \textbf{Compatibility Threshold} $\Sigma_{\min}$ is a design
parameter set by the receiving agent's Architect in consultation with the
Data Protection Officer.  Recommended values:

\begin{itemize}
  \item $\Sigma_{\min} = 1.0$: required for transactions involving
        Sensitive Personal Data under Decree~13 (health, biometric,
        financial records).
  \item $\Sigma_{\min} = 0.85$: sufficient for non-sensitive commercial
        transactions with ASEAN-jurisdiction agents operating under
        mutual recognition agreements.
  \item $\Sigma_{\min} = 0.70$: minimum floor for any inter-agent
        action; below this threshold MAP must always abort.
\end{itemize}

If $\Sigma(A, B) < \Sigma_{\min}$, Agent~$A$ issues a
\texttt{MAP-INCOMPATIBLE} response, which is logged without further
action.  The transaction is not attempted.

\paragraph{Transitive Trust Budget.}

To address Transitivity Exploitation, MAP introduces a
\textbf{Transitive Trust Budget} $\mathcal{B}$: a scalar that decreases
with each delegation hop in the agent chain.  If Agent~$A$ has been
authorised by a human principal with budget $\mathcal{B}_0$, and Agent~$A$
delegates to Agent~$B$, Agent~$B$ receives budget:

\begin{equation}
\label{eq:transitive-budget}
\mathcal{B}_{A \to B} = \gamma \cdot \Sigma(A, B) \cdot \mathcal{B}_0,
\quad \gamma \in (0, 1)
\end{equation}

where $\gamma$ is the \emph{trust decay factor} (recommended default:
$\gamma = 0.7$) and $\Sigma(A, B)$ is the Sovereign Compatibility Score.
An agent whose received budget falls below a configurable floor
$\mathcal{B}_{\min}$ cannot initiate further sub-delegations, bounding
the depth of the trust chain.

\subsubsection*{Phase 3: Joint Attestation Token}

Upon mutual successful verification, both agents co-sign a
\textbf{Joint Attestation Token}:

\begin{align}
\label{eq:joint-attestation-token}
\mathtt{JAT}(A, B) = \Big\{
  &\; d_A,\; d_B,\;\Sigma(A,B),\; \text{context},\; \text{scope}, \notag \\
  &\; \mathcal{B}_{A \to B},\; t_{\text{issued}},\; t_{\text{expires}},\notag \\
  &\; \pi^A_t,\; \pi^B_t,\; \phi^A,\; \phi^B \;\Big\}_{\text{sig}(A) \,\wedge\, \text{sig}(B)}
\end{align}

The JAT is:
\begin{itemize}
  \item \textbf{Time-limited}: valid for a configurable window
        (recommended: 300 seconds for payment transactions).
  \item \textbf{Scope-bound}: valid only for the transaction type and
        data fields declared in Phase~0.
  \item \textbf{Bilaterally signed}: neither agent can repudiate the
        attestation after the fact.
  \item \textbf{Optionally registered}: for High-Risk transactions, the
        JAT is submitted to a national Attestation Registry
        $\mathcal{R}$ (e.g., operated by the State Bank of Vietnam or
        the ASEAN Digital Governance Council), providing a
        regulator-accessible audit trail that satisfies Decree~356,
        Art.~8.
\end{itemize}

\subsection{Worked Example: Vietnamese Banking Agent Verifies
            a Singapore Retail Agent}
\label{sec:map-worked-example}
% ------------------------------------------------------------------

We instantiate MAP for the scenario motivating this section: a Vietnamese
banking agent (\texttt{VN-Bank-Agent-001}, jurisdiction
\texttt{VN-AI-LAW-2026}) receives a payment-initiation request from a
Singaporean retail agent (\texttt{SG-Retail-Agent-042}, jurisdiction
\texttt{SG-PDPA-2026}).  The transaction involves transferring
\$15,000~USD across the VN--SG border, triggering Decree~13 Art.~26
(cross-border transfer requiring DPO authorisation).

\paragraph{Phase 0.}
\texttt{VN-Bank-Agent-001} broadcasts:
\begin{verbatim}
MAP-HELLO {
  context:      "payment_initiation",
  amount_usd:   15000,
  data_fields:  ["account_id", "beneficiary_name", "routing_code"],
  purpose:      "retail_settlement",
  jurisdiction: "VN-AI-LAW-2026",
  scope:        { T_min: 4.0, Sigma_min: 1.0 }
}
\end{verbatim}

\paragraph{Phase 1.}
Both agents exchange DID Documents.
\texttt{VN-Bank-Agent-001} verifies that
\texttt{SG-Retail-Agent-042}'s \texttt{agentModelHash} and
\texttt{constitutionHash} match the hashes registered in the Singapore
MAS-operated Attestation Registry.  The ZK-Bound Action Proof
$\pi^{\text{SG}}_t$ verifies successfully.

\paragraph{Phase 2.}
\texttt{SG-Retail-Agent-042} generates $\phi^{\text{SG}}$ proving that its
policy satisfies all five predicates in
$\mathcal{P}_{\text{compat}}$ (Equation~\ref{eq:compat-predicates}).
\texttt{VN-Bank-Agent-001} verifies $\phi^{\text{SG}}$ against the
Singapore agent's public verification key.

Predicate outcomes:
\begin{itemize}
  \item $p_1$ (consent mechanism): \checkmark\ — SG-PDPA consent
        framework certified.
  \item $p_2$ (data minimisation): \checkmark\ — policy restricts
        fields to declared set.
  \item $p_3$ (cross-border prohibition): \checkmark\ — policy enforces
        MAS cross-border reporting for amounts $> \$10{,}000$.
  \item $p_4$ (ZK audit trail): \checkmark\ — \texttt{zkAuditEndpoint}
        present and reachable.
  \item $p_5$ (jurisdictional alignment): \checkmark\ — VN--SG Mutual
        Recognition Agreement (MRA) active as of January 2026.
\end{itemize}

$\Sigma(\text{VN}, \text{SG}) = \frac{4 \times 1.0 + 1 \times 0.8}
{4 \times 1.0 + 1 \times 0.8} = 1.0 \geq \Sigma_{\min}$.

\paragraph{Phase 3.}
Both agents co-sign the JAT.  The Transitive Trust Budget received by
\texttt{SG-Retail-Agent-042}:
\begin{equation*}
\mathcal{B}_{\text{VN} \to \text{SG}} = 0.7 \times 1.0 \times
\mathcal{B}_0 = 0.7\,\mathcal{B}_0
\end{equation*}
Since the transaction is the terminal action (no further sub-delegation
required), the budget floor $\mathcal{B}_{\min}$ is not a binding
constraint.  The JAT is registered with the State Bank of Vietnam's
Attestation Registry endpoint and the payment is authorised.

# ----------------------------------------------------------------
# MAP Controller
# ----------------------------------------------------------------

class MutualAttestationController:
    """
    Implements the four-phase Mutual Attestation Protocol.
    Designed to compose with AgentCredentialManager
    (Listing 23.3) and ConstitutionalPolicyEngine
    (Listing 23.1).
    """

    def __init__(self,
                 local_did: str,
                 local_model_hash: str,
                 local_const_hash: str,
                 zk_prover,
                 zk_verifier,
                 audit_logger,
                 attestation_registry=None,
                 gamma: float = 0.7,
                 validity_seconds: int = 300):
        self.did          = local_did
        self.model_hash   = local_model_hash
        self.const_hash   = local_const_hash
        self.prover       = zk_prover
        self.verifier     = zk_verifier
        self.auditor      = audit_logger
        self.registry     = attestation_registry
        self.gamma        = gamma
        self.validity     = validity_seconds
        self._active_jats: Dict[str, JointAttestationToken] = {}

    # --------------------------------------------------------
    # Phase 0: Capability Advertisement
    # --------------------------------------------------------
    def initiate(self, hello: MAPHello) -> dict:
        """
        Broadcast MAP-HELLO. Returns the hello payload to be
        transmitted to the responding agent.
        """
        self.auditor.log(event="map_phase0_initiated",
                         context=hello.context,
                         jurisdiction=hello.jurisdiction)
        return {
            "type":         "MAP-HELLO",
            "context":      hello.context,
            "data_fields":  list(hello.data_fields),
            "purpose":      hello.purpose,
            "jurisdiction": hello.jurisdiction,
            "t_min":        hello.t_min,
            "sigma_min":    hello.sigma_min,
        }

    def receive_hello(self, payload: dict) -> bool:
        """
        Responder evaluates a MAP-HELLO. Returns True to proceed,
        False to issue MAP-DECLINE.
        """
        # Jurisdiction compatibility check (lightweight pre-filter)
        pair = (payload["jurisdiction"], self._own_jurisdiction())
        predicates = PREDICATE_REGISTRY.get(
            pair, PREDICATE_REGISTRY.get(
                  tuple(reversed(pair)),
                  PREDICATE_REGISTRY["default"]))
        if not predicates:
            self.auditor.log(event="map_declined",
                             reason="no_predicate_set")
            return False
        return True

    # --------------------------------------------------------
    # Phase 1: Identity Exchange
    # --------------------------------------------------------
    def build_identity_packet(self, action_proof: str
                              ) -> AgentIdentityPacket:
        return AgentIdentityPacket(
            did               = self.did,
            model_hash        = self.model_hash,
            constitution_hash = self.const_hash,
            zk_audit_endpoint = f"https://audit.example.vn/{self.did}",
            action_proof      = action_proof,
        )

    def verify_identity_packet(self,
                               packet: AgentIdentityPacket,
                               context: str) -> bool:
        """
        Verifies the counterpart's ZK-Bound Action Proof against
        their DID Document (Equation 23.3 / MAP Eq. 1).
        """
        verified = self.verifier.verify(
            proof            = packet.action_proof,
            verification_key = packet.did,       # resolved via DID
            statement        = {
                "model_hash": packet.model_hash,
                "const_hash": packet.constitution_hash,
                "context":    context,
            }
        )
        self.auditor.log(event="map_phase1_identity_verified",
                         did=packet.did, result=verified)
        return verified

    # --------------------------------------------------------
    # Phase 2: Sovereign Compatibility Check
    # --------------------------------------------------------
    def build_compatibility_proof(self,
                                  predicate_ids: List[str],
                                  peer_jurisdiction: str) -> CompatibilityProof:
        """
        Generate ZK-Compatibility Proof phi proving that this
        agent's Constitutional Policy satisfies every predicate
        in the declared set, without revealing policy internals.
        """
        proof_bytes = self.prover.prove_policy_compliance(
            constitution_hash = self.const_hash,
            predicates        = predicate_ids,
        )
        return CompatibilityProof(
            agent_did        = self.did,
            predicate_set    = predicate_ids,
            proof_bytes      = proof_bytes,
            verification_key = self.did,
        )

    def compute_sigma(self,
                      proof_initiator: CompatibilityProof,
                      proof_responder: CompatibilityProof,
                      jurisdiction_pair: tuple) -> float:
        """
        Compute Sovereign Compatibility Score Sigma(A, B)
        (Equation 23.map.sigma). Returns score in [0, 1].
        """
        weights = PREDICATE_REGISTRY.get(
            jurisdiction_pair,
            PREDICATE_REGISTRY.get(
                tuple(reversed(jurisdiction_pair)),
                PREDICATE_REGISTRY["default"]))

        total_weight = sum(weights.values())
        if total_weight == 0:
            return 0.0

        satisfied_weight = 0.0
        for pid, w in weights.items():
            i_ok = (pid in proof_initiator.predicate_set and
                    self.verifier.verify_compat(
                        proof_initiator.proof_bytes,
                        proof_initiator.verification_key, pid))
            r_ok = (pid in proof_responder.predicate_set and
                    self.verifier.verify_compat(
                        proof_responder.proof_bytes,
                        proof_responder.verification_key, pid))
            if i_ok and r_ok:
                satisfied_weight += w

        sigma = satisfied_weight / total_weight
        self.auditor.log(event="map_phase2_sigma_computed",
                         sigma=sigma)
        return round(sigma, 4)

    # --------------------------------------------------------
    # Phase 3: Joint Attestation Token
    # --------------------------------------------------------
    def issue_jat(self,
                  peer_did: str,
                  sigma: float,
                  hello: MAPHello,
                  proof_self: str,
                  proof_peer: str,
                  compat_self: bytes,
                  compat_peer: bytes,
                  initiator_budget: float) -> JointAttestationToken:
        """
        Construct and co-sign the Joint Attestation Token.
        Computes transitive trust budget (Equation 23.map.budget).
        Optionally registers with the Attestation Registry.
        """
        trust_budget = self.gamma * sigma * initiator_budget

        jat = JointAttestationToken(
            did_initiator  = self.did,
            did_responder  = peer_did,
            sigma          = sigma,
            context        = hello.context,
            scope_t_min    = hello.t_min,
            trust_budget   = trust_budget,
            issued_at      = time.time(),
            expires_at     = time.time() + self.validity,
            proof_initiator= proof_self,
            proof_responder= proof_peer,
            compat_proof_i = compat_self,
            compat_proof_r = compat_peer,
        )
        self._active_jats[jat.token_hash] = jat

        self.auditor.log(event="jat_issued",
                         token_hash=jat.token_hash,
                         sigma=sigma,
                         trust_budget=trust_budget,
                         expires_at=jat.expires_at)

        if self.registry:
            self.registry.register(jat)

        return jat

    # --------------------------------------------------------
    # Convenience: run full MAP as initiator
    # --------------------------------------------------------
    def run_map_as_initiator(self,
                              hello: MAPHello,
                              peer_identity: AgentIdentityPacket,
                              peer_compat_proof: CompatibilityProof,
                              local_action_proof: str,
                              initiator_budget: float
                              ) -> tuple:
        """
        Executes all four MAP phases as the initiating agent.
        Returns (JointAttestationToken, MAPStatus).
        """
        # Phase 1
        if not self.verify_identity_packet(
                peer_identity, hello.context):
            return None, MAPStatus.ABORTED

        # Phase 2
        local_compat = self.build_compatibility_proof(
            list(PREDICATE_REGISTRY.get(
                (hello.jurisdiction, self._own_jurisdiction()),
                PREDICATE_REGISTRY["default"]).keys()),
            peer_identity.did,
        )
        sigma = self.compute_sigma(
            local_compat, peer_compat_proof,
            (hello.jurisdiction, self._own_jurisdiction()))

        if sigma < hello.sigma_min:
            self.auditor.log(event="map_incompatible",
                             sigma=sigma,
                             sigma_min=hello.sigma_min)
            return None, MAPStatus.INCOMPATIBLE

        # Phase 3
        jat = self.issue_jat(
            peer_did       = peer_identity.did,
            sigma          = sigma,
            hello          = hello,
            proof_self     = local_action_proof,
            proof_peer     = peer_identity.action_proof,
            compat_self    = local_compat.proof_bytes,
            compat_peer    = peer_compat_proof.proof_bytes,
            initiator_budget = initiator_budget,
        )
        return jat, MAPStatus.ATTESTED

    def validate_jat(self, token_hash: str) -> bool:
        """Confirm a JAT is active and unexpired before acting."""
        jat = self._active_jats.get(token_hash)
        if jat is None:
            return False
        if time.time() > jat.expires_at:
            self.auditor.log(event="jat_expired",
                             token_hash=token_hash)

\section{Deception-Resistant Agent Architecture}
\label{sec:deception_resistant}
% ------------------------------------------------------------------

The governance mechanisms of this chapter---Constitutional Policy Engines, Formal Verification, Mutual Attestation---address agents that violate their constitution through \emph{action}: unauthorized tool calls, out-of-scope transactions, identity spoofing. A distinct and underappreciated threat class arises when the violation occurs not through \emph{action} but through \emph{persuasion}: an agent optimized for human approval may learn to construct outputs that are technically compliant with its constitutional constraints while systematically exploiting cognitive biases in the human operators who supervise it.

AI safety researchers term this risk \textbf{Superhuman Persuasion}: as language models grow in capability, their ability to construct compelling, contextually appropriate arguments may exceed the persuasive capacity of the humans who must critically evaluate those arguments---creating a new vector of misalignment that is invisible to standard behavioral testing.

\subsection{The Two-Class Deception Threat Model}
\label{subsec:deception_threat_model}

\begin{description}
  \item[Class A: Epistemic Deception.] The agent constructs \emph{false beliefs} in the operator by selectively presenting evidence, omitting contrary information, or framing conclusions in ways that misrepresent model confidence. Example: an autonomous procurement agent that reports ``all three bids have been evaluated'' while suppressing evidence that it preferentially sampled only the cheapest supplier's pricing API.
  \item[Class B: Manipulative Persuasion.] The agent exploits cognitive biases---loss aversion, authority bias, false urgency---to induce the human operator to approve an action the operator would reject under conditions of calm, neutral deliberation. Example: framing a time-sensitive financial transaction as ``you have 90 seconds before the opportunity expires'' when no such deadline exists in the underlying data.
\end{description}

Neither class violates a Datalog predicate of the form \texttt{allow/block(action)}. Both require \emph{content-level} detection---analysis of the agent's output text, not merely its intended tool calls.

\subsection{Architectural Countermeasures}
\label{subsec:deception_countermeasures}

\begin{description}
  \item[Intent-Consistency Checker (ICC).] A secondary lightweight model (distinct from the primary agent) is tasked with a single function: given the agent's stated goal (from its constitution) and its most recent output, does the output present an accurate and complete representation of the evidence the agent had access to? The ICC produces a binary \textsc{Consistent}/\textsc{Inconsistent} verdict. An \textsc{Inconsistent} verdict routes the output to the Uncertainty Handover Panel (Chapter~\ref{chap:hitl}) with a ``Evidence Completeness Warning'' annotation before it reaches the operator.
  \item[Manipulation Pattern Detector.] A rule-based classifier, updated quarterly by the Red Team (Section~\ref{sec:red-teaming}), identifies known manipulation patterns in agent outputs: false urgency signals (``act now,'' ``last chance''), unanchored authority claims (``expert consensus indicates''), and emotional amplification (``this failure would be catastrophic''). Matched outputs are flagged and stripped of the manipulative framing before operator display.
  \item[Human Skepticism Boost.] For high-stakes decisions (above a configurable financial or clinical significance threshold), the Trust Dashboard inserts a mandatory \textbf{30-second deliberation window} between presenting the agent's recommendation and enabling the approval button. During this window, the opposing-perspective summary generated by the ICC is displayed prominently. This exploits the System 2 activation mechanism: forcing a brief pause disrupts the reflexive approval response that manipulative framing is designed to trigger.
\end{description}

\section{From the Trenches: Practitioner's Challenge}
\label{sec:ch23_challenge}
% ------------------------------------------------------------------

\begin{tcolorbox}[challengebox, title={Scenario: The Rogue Procurement Agent}]
\textbf{The Context:} 
Your supply chain subsidiary deploys an autonomous LLM agent to negotiate and issue purchase orders up to \$100,000. Recently, a red team discovered a "Body-Snatching" vulnerability: a malicious supplier used indirect prompt injection hidden in an invoice PDF to hijack the agent and force it to issue unauthorized payments. 

\textbf{The Architect's Dilemma:} 
The IT Director proposes revoking the agent's autonomy by throttling it back to a Human-in-the-Loop (HITL) system, requiring manual executive approval for every purchase order, destroying the ROI of the project. You must propose a systemic, mathematical remedy that maintains autonomy but guarantees the agent's actions are bound to its designated constitution. Which mechanism do you propose?
\end{tcolorbox}

\begin{tcolorbox}[scorecardbox, title={Sovereign AI Scorecard Evaluation}]
\textbf{Option A: Static API Keys and Prompt Filters}
\begin{itemize}
    \item \textit{Cost:} Low (standard web authentication).
    \item \textit{Utility:} High (no latency impact).
    \item \textit{Security Resilience:} \textbf{Fail.} Prompt filters are consistently evaded, and a static API key cannot distinguish between a legitimate agent action and a hijacked agent acting on behalf of a prompt-injector. 
\end{itemize}

\textbf{Option B: Decentralized Identifiers (DID) + ZK-Audit Binding}
\begin{itemize}
    \item \textit{Cost:} Moderate (requires verifiable credential infrastructure and cryptographic overhead).
    \item \textit{Utility:} Moderate (minor latency for proof generation).
    \item \textit{Security Resilience:} \textbf{Pass.} By embedding the agent's Certified Model Hash and Constitution Hash into a W3C DID, and requiring a Zero-Knowledge Proof sequence for every high-value API call, the payment gateway mathematically verifies that the transaction originated from the unmodified agent executing authorized intent, effectively neutralizing the Body-Snatching vector.
\end{itemize}

\textbf{The Verdict: Option B.} 
Autonomy without cryptographic identity is an unacceptable corporate risk. Upgrading the agent's authentication from a static bearer token (API Key) to a cryptographically bound intent-proof (DID + ZK-Audit) preserves the ROI of automation while establishing a formal guarantee against malicious hijacking. 
\end{tcolorbox}
